---
title: Pod 帯域幅制限の設定
description: モデルダウンロード Pod と通常ジョブ Pod のネットワーク上限を個別に設定します
---

管理者はプラットフォーム設定の「Pod 帯域幅制限」から、モデルダウンロード、通常ジョブの受信、通常ジョブの送信帯域幅を個別に管理できます。bandwidth CNI が未導入のクラスターへ影響しないよう、この機能は既定で無効です。

## クラスター機能の確認

画面は次の条件をリアルタイムで確認します。

- `kube-flannel-cfg` の `cni-conf.json` に `bandwidth` プラグインと `bandwidth` capability が含まれていること。
- Flannel DaemonSet に bandwidth binary をインストールする initContainer があること。
- Flannel DaemonSet のロールアウトがすべての対象ノードで完了し、Ready であること。

いずれかの条件を満たさない場合、プラットフォームは制限の有効化を拒否します。有効化後にクラスター機能が失われた場合も、起動できない Pod を作成しないよう、新しい帯域制限付きジョブの送信を停止します。

## 設定項目

- **モデルダウンロード帯域幅**：model-dl Pod の受信 annotation を設定し、ダウンロードデータが Pod に入る速度を制限します。
- **通常ジョブの受信帯域幅**：各通常ジョブ Pod に入るデータ速度を制限します。
- **通常ジョブの送信帯域幅**：各通常ジョブ Pod から出るデータ速度を制限します。

画面では数値と単位を分けて入力し、Kbit/s、Mbit/s、Gbit/s から単位を選択できます。たとえば数値 `1` と Gbit/s は Kubernetes 数量 `1G` として保存されます。

設定はシステム設定に保存され、バックエンドを再起動せずに新しい Pod へ反映されます。CNI は Pod sandbox 作成時にのみ annotation を読み取るため、実行中の Pod は変更されず、新しい値を適用するには再作成が必要です。

## 検証時の注意

containerd は bandwidth CNI に約 512 MiB の初期 burst を渡すことがあります。小さな転送は burst 内で完了し、安定時の制限が見えない場合があります。検証には 512 MiB を十分に超えるデータを使用してください。
