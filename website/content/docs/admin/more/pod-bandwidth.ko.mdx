관리자는 플랫폼 설정의 **Pod 대역폭 제한**에서 모델 다운로드, 일반 작업 인바운드, 일반 작업 아웃바운드 대역폭을 각각 관리할 수 있습니다. bandwidth CNI가 설치되지 않은 클러스터에 영향을 주지 않도록 이 기능은 기본적으로 비활성화되어 있습니다.

## 클러스터 기능 확인

페이지는 다음 조건을 실시간으로 확인합니다.

- `kube-flannel-cfg`의 `cni-conf.json`에 `bandwidth` 플러그인과 `bandwidth` capability가 포함되어 있어야 합니다.
- Flannel DaemonSet에 bandwidth binary를 설치하는 initContainer가 있어야 합니다.
- Flannel DaemonSet 롤아웃이 모든 대상 노드에서 완료되고 Ready 상태여야 합니다.

조건 중 하나라도 충족되지 않으면 플랫폼은 제한 활성화를 거부합니다. 기능을 활성화한 후 클러스터 지원이 사라진 경우에도 시작할 수 없는 Pod를 만들지 않도록 새로운 대역폭 제한 작업 제출을 차단합니다.

## 설정 항목

- **모델 다운로드 대역폭**: model-dl Pod의 인바운드 annotation을 설정하고 다운로드 데이터가 Pod로 들어오는 속도를 제한합니다.
- **일반 작업 인바운드**: 각 일반 작업 Pod로 들어오는 데이터 속도를 제한합니다.
- **일반 작업 아웃바운드**: 각 일반 작업 Pod에서 전송되는 데이터 속도를 제한합니다.

페이지에서는 숫자와 단위를 분리해 입력하며 Kbit/s, Mbit/s, Gbit/s 중에서 선택할 수 있습니다. 예를 들어 값 `1`과 Gbit/s는 Kubernetes 수량 `1G`로 저장됩니다.

설정은 시스템 구성에 저장되며 백엔드를 다시 시작하지 않고 새 Pod에 적용됩니다. CNI는 Pod sandbox를 생성할 때만 annotation을 읽으므로 실행 중인 Pod는 변경되지 않으며 새 값을 사용하려면 재생성해야 합니다.

## 검증 참고 사항

containerd는 bandwidth CNI에 약 512 MiB의 초기 burst를 제공할 수 있습니다. 작은 전송은 burst 안에서 끝나 안정 상태의 제한이 보이지 않을 수 있습니다. 검증할 때는 512 MiB보다 충분히 큰 데이터를 사용하세요.
