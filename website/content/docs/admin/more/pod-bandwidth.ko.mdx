관리자는 플랫폼 설정의 **Pod 대역폭 제한**에서 모델 다운로드와 Volcano 작업의 인바운드 및 아웃바운드 대역폭을 각각 관리할 수 있습니다. 작업 설정은 일반 작업과 백필 작업에 모두 적용됩니다. bandwidth CNI가 설치되지 않은 클러스터에 영향을 주지 않도록 이 기능은 기본적으로 비활성화되어 있습니다.

## 클러스터 기능 확인

페이지는 다음 조건을 실시간으로 확인합니다.

- `kube-flannel` namespace에 `kube-flannel-cfg` ConfigMap과 `kube-flannel-ds` DaemonSet이 있어야 합니다.
- ConfigMap의 `cni-conf.json`에 `bandwidth` 플러그인과 `bandwidth` capability가 포함되어 있어야 합니다.
- DaemonSet에 `install-bandwidth-plugin` initContainer가 있어야 하며, 호스트 `/opt/cni/bin`을 마운트하고 `/opt/cni/bin/bandwidth`를 `/host/opt/cni/bin/bandwidth`로 복사해야 합니다.
- DaemonSet 롤아웃이 모든 대상 노드에서 완료되고 Ready 상태여야 합니다.

이는 Crater가 현재 지원하는 Flannel 배포 계약입니다. namespace, 리소스 이름 또는 설치 방식이 다르면 기능을 사용할 수 없는 것으로 표시됩니다. 조건을 충족하지 못하면 직접 제출된 작업은 생성 전에 중지되고 Prequeue 작업은 기능이 복구될 때까지 활성화 단계에서 대기합니다.

## 설정 항목

- **모델 다운로드 대역폭**: model-dl Pod의 인바운드 annotation을 설정하고 다운로드 데이터가 Pod로 들어오는 속도를 제한합니다.
- **작업 인바운드**: 일반 작업과 백필 작업 Pod로 들어오는 데이터 속도를 제한합니다.
- **작업 아웃바운드**: 일반 작업과 백필 작업 Pod에서 전송되는 데이터 속도를 제한합니다.

페이지에서는 숫자와 단위를 분리해 입력하며 Kbit/s, Mbit/s, Gbit/s 중에서 선택할 수 있습니다. 예를 들어 값 `1`과 Gbit/s는 Kubernetes 수량 `1G`로 저장됩니다.

설정은 시스템 구성에 저장되며 백엔드를 다시 시작할 필요가 없습니다. 모델 다운로드 Pod 또는 Volcano 작업을 실제 생성할 때 최신 설정을 읽고 Prequeue 작업은 제출 시점이 아닌 활성화 시점의 설정을 사용합니다. CNI는 Pod sandbox를 생성할 때만 annotation을 읽으므로 실행 중인 Pod는 재생성해야 합니다.

## 검증 참고 사항

containerd는 bandwidth CNI에 약 512 MiB의 초기 burst를 제공할 수 있습니다. 작은 전송은 burst 안에서 끝나 안정 상태의 제한이 보이지 않을 수 있습니다. 검증할 때는 512 MiB보다 충분히 큰 데이터를 사용하세요.
