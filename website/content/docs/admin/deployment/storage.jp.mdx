Crater では、すべてのデータに同じストレージシステムを使う必要はなく、すべてのクラスターが以下の全ストレージ機能を備える必要もありません。有効化するコンポーネントのアクセス方式、性能、信頼性の要件に合わせて組み合わせます。

標準の Kubernetes は PV、PVC、`StorageClass` などのストレージインターフェースを定義しますが、OpenEBS、NFS、Ceph を自動的に提供するものではありません。Crater Chart は PVC を宣言してコンポーネントにマウントしますが、これらのストレージバックエンドはインストールしません。Crater をデプロイする前に、管理者が必要なストレージシステムをデプロイまたは接続し、対応する `StorageClass` をクラスターに用意する必要があります。

## アーキテクチャ概要

アプリケーションが特定のストレージサーバーを直接選ぶことはありません。Kubernetes の共通インターフェースを通じてストレージを使用します。

```text
Crater コンポーネント → PVC → StorageClass → ストレージドライバーまたは provisioner → 実際のストレージバックエンド
```

Crater が重視するのはボリュームが必要なアクセスモードを満たすかどうかであり、底層に OpenEBS、NFS、Ceph のどれを使うかはクラスター管理者が決定します。以下の表は、必要に応じて組み合わせるストレージ機能を示したもので、すべてのデプロイに必須の固定層ではありません。

| ストレージ機能 | 必要となる場合 | 典型的なアクセス方式 | Crater での主な用途 | 一般的な実装 |
| --- | --- | --- | --- | --- |
| ノードローカルストレージ | BuildKit を有効にする場合、または本サイトの CloudNativePG デプロイ方法を採用する場合 | 単一ノードにマウント、低レイテンシ | 再生成可能なビルドキャッシュ、または上位アプリケーションが複製を保証する永続データ | OpenEBS LocalPV |
| 共有ファイルストレージ | Crater の基本インストールで通常必要 | 複数の Pod またはノードから同時アクセス（RWX） | ユーザーの HOME、データセット、モデル、共有ファイル、バックアップ | NFS または CephFS |
| 分散ブロックストレージ | 永続 RWO ボリュームが必要な独立コンポーネントを有効にする場合に選択 | 通常は単一ノードから読み書き（RWO） | 現在の Prometheus Stack デプロイガイドにおける履歴メトリクス | Ceph RBD |

PVC が `Bound` であることは、Kubernetes が要求にストレージを割り当てたことだけを意味します。対象ノードでマウントが成功したことまでは保証しません。実際に利用できるかどうかは、ノード側のマウントツール、CSI プラグイン、ネットワーク、ストレージバックエンドの状態にも依存します。

## ストレージ機能の組み合わせ

### ノードローカルストレージ

[OpenEBS LocalPV](https://openebs.io/docs/user-guides/localpv) は、ノード上のローカルディレクトリまたはディスクを Kubernetes の永続ボリュームとして公開します。LocalPV は Kubernetes がサポートするボリュームの種類ですが、動的プロビジョニングに使う OpenEBS provisioner は管理者が別途インストールする必要があります。本ドキュメントのデプロイ方法では `openebs-hostpath` StorageClass を使用します。

Crater はこの機能を次の 2 つの具体的な場面で使用します。

- BuildKit イメージビルドサービスを有効にすると、各 BuildKit インスタンスは個別の LocalPV にビルドキャッシュを保存します。キャッシュをローカルに置くことで、イメージレイヤーの重複ダウンロードやビルドを減らせます。キャッシュが失われると後続のビルドは遅くなりますが、データは再生成できます。BuildKit はデフォルトで無効のため、イメージビルドを使わない場合はこの目的の LocalPV は不要です。
- 本サイトの CloudNativePG デプロイ方法では、各 PostgreSQL インスタンスが LocalPV にデータベースファイルを保存します。ボリューム自体はデータを複製しないため、複製、フェイルオーバー、バックアップは CloudNativePG が担う必要があります。

LocalPV は分散ストレージではなく、データを他のノードへ自動的に複製しません。ボリュームにはノード制約があるため、Pod はデータを保持するノードに戻らなければ引き続き利用できません。再生成可能なデータ、または上位アプリケーションが冗長性を確保するデータにだけ使用してください。

### 共有ファイルストレージ

storage-server、ユーザーの共有ファイル、データベースバックアップは、複数の Pod またはノードから同じディレクトリにアクセスする必要があります。通常は `ReadWriteMany` をサポートする NFS または CephFS を使用します。Crater における役割は同じですが、実装と運用コストは異なります。

- **NFS** を使う場合、管理者は NFS サーバー、ノード上の NFS クライアント機能、Kubernetes provisioner を用意する必要があります。比較的導入が容易で、信頼できる NFS サービスが既にあるクラスターや小規模のクラスターに適しています。容量、性能、可用性は NFS サーバーに依存します。
- **CephFS** を使う場合、管理者は Ceph をデプロイまたは接続し、Rook や CephFS CSI ドライバーなどの Kubernetes 統合コンポーネントをインストールする必要があります。既に Ceph を運用し、拡張性とストレージ側の耐障害性を必要とするクラスターに適しますが、導入と運用のコストは高くなります。

Crater のコア Chart は、デフォルトで `nfs` という名前の StorageClass を使って共有ボリュームを作成し、同じ PVC を複数の storage-server レプリカにマウントしてユーザーファイルを保存・提供します。ユーザーが Jupyter、WebIDE、または学習ワークロードを起動すると、Crater はユーザーの HOME ディレクトリをコンテナーにマウントします。ユーザーが選択した Account または公開領域のファイル、データセット、モデルも、このストレージからワークロードコンテナーにマウントされます。これらのマウントは、Crater に設定された RWX 共有 PVC、または同じ底層ストレージを使う任意の読み取り専用 PVC を使います。

通常のユーザーイメージはコンテナーの実行環境を提供するだけで、これらのファイルが LocalPV、NFS、CephFS のどれを使うかは決定しません。実際のバックエンドは、共有 PVC が参照する `StorageClass` によって決まります。データベースバックアップ機能も、デフォルトで `nfs` StorageClass を使う別の RWX ボリュームにバックアップを書き込みます。

ここでの `nfs` はデフォルトの StorageClass 名にすぎず、Crater が NFS をインストールすることを意味しません。クラスターで CephFS を使う場合は、この 2 か所を実際の CephFS `StorageClass` に変更できます。Crater によるボリュームの使い方は変わりません。

### 分散ブロックストレージ

[Ceph RBD](https://rook.io/docs/rook/latest/ceph-block.html) は Ceph クラスターから仮想ブロックデバイスを提供し、Kubernetes がノード上でフォーマットして Pod にマウントします。CephFS と同様に、Ceph クラスター、Rook、RBD CSI ドライバーは管理者が事前にデプロイする必要があり、Crater が自動的にインストールすることはありません。

Crater のコア Chart はデフォルトで RBD を直接使用しないため、分散ブロックストレージは Crater の基本依存関係ではありません。本サイトの Prometheus Stack デプロイ方法では、`rook-ceph-rbd` を使って Prometheus の履歴メトリクスを保存します。Prometheus には信頼できる永続ボリュームが必要ですが、複数ノードが同じファイルシステムに同時に書き込む必要はありません。管理者が別途 Grafana の永続化を有効にする場合、Grafana にも適切な永続ボリュームが必要ですが、RBD である必要はありません。

Ceph RBD がなくても、共有ファイルストレージが利用できれば、ユーザーの HOME、データセット、モデル、通常のジョブには影響しません。Prometheus Stack をデプロイする場合は、RBD を要件を満たす他の RWO `StorageClass` に置き換えられます。Prometheus に永続ボリュームを提供しない場合、Pod の再作成時に履歴メトリクスが失われる可能性があります。PVC を宣言しても利用可能な `StorageClass` がなければ、対象 Pod はデプロイを完了できません。

RBD と CephFS は同じ Ceph クラスターを底層で共有できますが、異なるボリュームです。RBD はブロックデバイスを提供し、CephFS は共有ファイルシステムを提供し、それぞれ別の CSI ドライバーを使用します。RBD は storage-server が必要とする RWX ファイルストレージの代わりにはならず、ノードが RBD をサポートすることは CephFS もサポートすることを意味しません。

したがって、NFS のみを共有ファイルストレージとして使うクラスターでは、永続 RWO ボリュームを必要とするコンポーネントを有効にしなければ、分散ブロックストレージを導入する必要はありません。NFS と RBD は同じ用途の互換製品ではありません。クラスターは NFS のみを使うことも、必要に応じて NFS と RBD を併用することもできます。

## Crater でのストレージ利用

Crater のコア Chart は、次の設定で既存の StorageClass に接続します。

| 用途 | 使用する場合 | Chart 設定 | 要件 | デフォルト |
| --- | --- | --- | --- | --- |
| ユーザー HOME、共有ファイル、データセット、モデル | 基本インストール。storage-server とユーザーワークロードが共用 | `storage.storageClass` | RWX | `nfs` |
| PostgreSQL バックアップボリューム | データベースバックアップと PVC 作成を有効にした場合 | `dbBackup.config.storageClass` | RWX | `nfs` |
| AMD BuildKit キャッシュ | AMD BuildKit を有効にした場合 | `buildkitConfig.amdConfig.cache.storageClass` | ノードローカル RWO ボリューム | `openebs-hostpath` |
| ARM BuildKit キャッシュ | ARM BuildKit を有効にした場合 | `buildkitConfig.armConfig.cache.storageClass` | ノードローカル RWO ボリューム | `openebs-hostpath` |

CloudNativePG や Prometheus などの独立コンポーネントは、それぞれのデプロイ設定でストレージを選択します。現在のドキュメントでは CloudNativePG に OpenEBS LocalPV、Prometheus に Ceph RBD を使用します。これらは記載されたデプロイ方法であり、Crater のコアコンポーネントが特定のストレージ製品を必須とするものではありません。管理者はクラスターの信頼性設計に合わせて調整できますが、各コンポーネントが必要とするアクセスモードは維持してください。

すべてのバックエンドをインストールする必要はありませんが、有効にする Crater 機能が必要とするストレージは用意する必要があります。基本インストールでは通常、storage-server とユーザーワークロード用の RWX `StorageClass` から始めます。BuildKit を有効にする場合、または本サイトの CloudNativePG デプロイを採用する場合は OpenEBS を導入します。本サイトの Prometheus デプロイには、Ceph RBD または他の適切な永続 RWO バックエンドを用意します。既に Ceph を運用しているクラスターでは、デフォルトの NFS を CephFS に置き換えることもできます。

大規模クラスターでは、ローカルキャッシュが共有ストレージを消費しないようにし、ノード間アクセスや障害復旧が必要なデータを共有ストレージに置く混合構成が一般的です。

## デプロイと運用の確認

Crater をデプロイする前、またはステートフルコンポーネントを更新する前に、少なくとも次を確認してください。

1. 実際のストレージバックエンドがデプロイまたは接続され、参照するすべての `StorageClass` が存在し、アクセスモードがコンポーネントの要件を満たしていること。
2. LocalPV の対象ノードにローカルパスが用意され、Pod の配置がボリュームのノード制約と一致していること。
3. NFS クライアント機能または対応する CSI ノードプラグインが、ワークロードを実行する可能性のあるすべてのノードをカバーしていること。
4. テスト用の PVC と Pod を作成し、対象ノードで実際にマウントと読み書きが成功すること。PVC が `Bound` であることだけで判断しないでください。

### CephFS CSI ノードカバレッジ

taint のあるコントロールプレーンノードやその他の専用ノードにワークロードをスケジュールできる場合、CephFS CSI DaemonSet もそれらのノードで実行され、ドライバーを登録できる必要があります。ワークロードの toleration は CSI プラグインに自動的に引き継がれません。

実際の障害では、storage-server は node selector と toleration を使い、`NoSchedule` taint のあるコントロールプレーンノードで実行されていましたが、Rook が管理する CephFS CSI DaemonSet には同じ toleration がありませんでした。そのため、ワークロード Pod は対象ノードにスケジュールできても、そのノードに CephFS CSI Pod がなく、`CSINode` に CephFS ドライバーも登録されていませんでした。

次の現象が組み合わせて発生した場合は、アプリケーションイメージより先に CSI ノードカバレッジと toleration を確認してください。

- 新規作成された Pod、または rollout で作成された Pod がノードにスケジュール済みなのに `ContainerCreating` のままで、コンテナーはまだ起動していない。
- PVC は `Bound` で、VolumeAttachment が `attached=true` を示していても、Pod イベントは `rook-ceph.cephfs.csi.ceph.com` が登録済み CSI ドライバーの一覧にないと報告する。
- 同じワークロードの既存 Pod は読み書きできるが、新規作成、移動、または再作成された Pod はボリュームをマウントできない。

古い Pod が動作し続けることは矛盾ではありません。CSI ノードプラグインはマウントの確立、アンマウント、再公開を担いますが、すべてのファイル読み書きを中継するわけではありません。プラグインが消失した後も、過去に確立された CephFS カーネルマウントは動作し続けることがありますが、Pod の再作成、移動、ノードの再起動、または再マウント時に問題が表面化します。

トラブルシューティングでは、ワークロードの対象ノードの taint と toleration、CephFS CSI DaemonSet のノードカバレッジ、`CSINode` のドライバー登録を照合します。Operator が管理する DaemonSet を直接編集せず、使用中の Rook バージョンがサポートする永続的な設定で Operator の期待状態を修正してください。

修正後は CSI DaemonSet の `desired`、`updated`、`ready`、`available`、`unavailable` を監視し、その後にワークロード Pod のマウントイベントを確認します。RollingUpdate が停滞した場合は、最初に異常となった CSI Pod とそのノードのイベントを調べてください。`maxUnavailable` が小さい場合、1 ノードの障害で後続の更新がブロックされることがあり、更新待ちのすべてのノードに障害があることを意味しません。
