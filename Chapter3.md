# 第3章 面向智算平台的多智能体协作框架设计

本章旨在针对当前智算平台运维领域面临的任务复杂度高、异构数据庞杂、跨平台适配困难以及高风险操作难以管控等痛点，提出一种名为 Mops (Multi-Agents for Intelligent Computing Platform Operations) 的多智能体协作框架。该框架将复杂的运维任务分解为多个智能体角色的协同工作流，并通过结构化的状态机、工件流转机制与统一的上下文管理，实现智能化、自动化的运维闭环。此外，针对安全生产的核心诉求，本框架在工具体系、权限校验与人机协同（HITL）等方面进行了深度设计。

## 3.1 智算平台运维任务建模

智算平台作为支撑大规模AI训练与推理的核心基础设施，其运维对象涵盖了底层的算力节点、网络存储资源，以及上层的作业调度、镜像管理、数据集分发等诸多层面。为了构建一个具备普适性与高效性的多智能体运维框架，首先需要对智算平台的运维任务进行系统性建模。

### 3.1.1 运维对象与用户角色

**运维对象建模：**
在智算平台的运行过程中，产生着海量的异构数据。现有的平台数据库（DB）通常记录了平台的用户信息、配额管理、长期的历史作业流水等持久化数据。而动态运行的AI作业（AI Jobs）则涉及实时的运行状态、计算节点分配、显存占用、日志输出以及事件流。这些对象构成了多智能体系统的“操作实体”。我们将运维对象抽象为四类核心实体：
1. **资源实体（Resource Entities）：** 包括 GPU/CPU 计算节点、存储卷、网络拓扑等。
2. **作业实体（Job Entities）：** 包括单机训练作业、分布式训练任务、推理服务部署等。
3. **策略实体（Policy Entities）：** 包含调度队列配置、资源配额策略、计费模型等。
4. **日志与事件实体（Log & Event Entities）：** 包括系统告警、容器标准输出、节点异常事件等。

**用户角色建模：**
不同的用户在平台中扮演着不同角色，具有差异化的操作意图与权限边界。本框架将用户角色抽象为：
1. **普通开发者/算法工程师：** 关注自身AI作业的运行状态、排队情况以及调试日志。其运维意图通常为“诊断我的作业为何失败”或“查看可用资源”。
2. **平台管理员/运维工程师：** 关注全局资源水位、节点健康度、异常任务干预以及配额调整。其意图倾向于“封锁故障节点”、“终止异常消耗任务”或“全局巡检”。
3. **审计人员：** 关注高危操作的历史记录、合规性与安全性溯源。

### 3.1.2 任务类型、风险等级与数据依赖

为了指导智能体的行为边界，我们对运维任务进行了多维度的分类。

**任务类型：**
1. **信息查询与问答（Information Retrieval & QA）：** 依赖平台使用手册与元数据，解答如“如何配置MPI环境变量”等问题。
2. **状态诊断与排障（Diagnostic & Troubleshooting）：** 例如“作业调度失败排查”，需要结合作业详情、节点状态与历史调度事件进行综合推理。
3. **状态变更与控制（State Mutation & Control）：** 涉及资源创建、任务终止、配置修改等写操作。

**风险等级划分：**
框架将所有工具与操作映射至三个风险等级：
- **低风险（Low Risk）：** 只读查询、日志获取、状态探测等，不会改变系统状态。
- **中风险（Medium Risk）：** 影响单个用户或单次作业的操作，如重启自身作业、修改自身配置等。
- **高风险（High Risk）：** 影响全局资源、其他用户作业或底层基础设施的操作，如节点下线、配额清零、批量终止作业等。

**数据来源与依赖获取机制：**
运维数据的获取必须考虑时效性与完整性。本框架通过工具抽象层（Tool Layer）定义了不同的数据获取实现：
- **实时API层：** 用于获取当前正在运行或排队中的全量作业列表（如运行中、排队中、近期失败的作业状态），提供高时效性的状态切片。
- **持久化DB层：** 用于支撑长周期的大规模数据统计与关联分析，例如近90天内的作业资源消耗趋势。DB层通过带有 `limit` 限制的分页查询提供服务，避免全量检索导致数据库过载。
- **真实平台日志与监控采集：** 对于告警信息，通过 Prometheus/Grafana 等监控系统的接口获取时序数据，经由转换脚本（如 `transform.py`）构造标准化的诊断上下文。

## 3.2 Mops框架总体架构设计

### 3.2.1 设计目标

**Mops (Multi-Agents for Intelligent Computing Platform Operations)** 旨在构建一个高度自动化且安全可控的智算运维大脑。其核心设计目标包括：
1. **意图精准捕获：** 能够结合当前页面上下文、用户权限与多轮对话历史，准确解析运维意图。
2. **专业化分工与协作：** 打破单一庞大模型的局限，引入角色化的智能体（如规划者、执行者），通过协作流降低单步幻觉。
3. **安全与可控（Guardrails）：** 在AI产生行动决策后、实际落盘前，引入强拦截机制与人机协同审批（Human-in-the-Loop）。
4. **跨平台可迁移：** 核心编排逻辑与具体平台的底层 API 解耦，通过适配器模式实现一套框架管理多套异构智算平台。

### 3.2.2 分层架构

Mops框架在逻辑上自底向上划分为四层：
1. **智算平台适配层（Platform Adapter Layer）：** 直接对接真实平台的API、数据库以及监控系统。负责将异构的接口封装为标准化的工具调用。
2. **工具与安全声明层（Tool & Security Declaration Layer）：** 定义各工具的入参、出参、功能描述与风险等级。实现基于角色的访问控制（RBAC）投影。
3. **多智能体协作与状态流转层（Multi-Agent Collaboration Layer）：** 框架的核心。包含意图路由（Intent Router）、协调器（Coordinator）以及各类执行智能体。通过多智能体系统状态（MASState）和工件（Artifacts）完成信息的无缝传递与阶段状态转换。
4. **多模态交互与上下文层（Interaction & Context Layer）：** 负责与用户进行SSE事件流通信，捕获前端页面的操作上下文，管理会话记忆与多轮状态。
## 3.3 多智能体角色设计与协作机制

在面对复杂的智算平台排障时，单一的大语言模型往往会因为上下文过载、规划能力不足或执行冲动而导致任务失败。为此，Mops框架设计了一套多角色分工的协作机制，将运维流程切分为规划、探索、执行与验证等独立阶段。

### 3.3.1 智能体角色定义

框架的核心智能体角色包括意图路由代理（Intent Agent）与工作流协调器（Coordinator），以及挂载于协调器下的四类核心工作节点和三类辅助节点。

**1. 意图路由代理 (Intent Agent)**
作为整个系统的流量入口，意图路由代理负责对用户的自然语言输入、页面上下文以及历史会话进行初步的分类与路由。为了兼顾效率与准确性，系统采用了启发式规则与轻量级LLM结合的两级路由算法（如算法1所示）。当检测到特定的页面上下文（如 `job_detail`）时，直接生成具有高置信度的诊断意图；对于复杂语义，则调用LLM进行意图槽位提取，最终决定进入何种子工作流。

**2. 协调代理 (Coordinator)**
协调器是整个系统的“大脑”。它不直接调用工具，而是负责维护全局的状态机，并决定当前任务应该流转到哪一个特定的阶段。协调器通过分析当前已有的共享状态（MASState）来做出决策（如算法2所示）。

**3. 核心工作角色**
- **规划器 (Planner)：** 负责接收高层意图，将宏观任务分解为有向无环图（DAG）或步骤列表形式的候选工具队列。
- **探索器 (Explorer)：** 负责信息收集（Information Gathering）。探索器仅被允许调用只读（Low Risk）工具。它采用迭代式证据收集机制（如算法5所示），逐步获取节点状态、作业日志，直到满足“证据充分”的条件。
- **执行器 (Executor)：** 专注执行状态变更操作（中高风险）。执行器必须严格遵循安全控制层的拦截规则，负责将规划中的参数组装并调用实际的写操作API。
- **验证器 (Verifier)：** 在执行器操作完毕后，验证器负责检查操作结果是否符合预期，或在规划器给出诊断结论后，进行对抗式验证（Adversarial Verification，如算法6所示），评估证据与结论的逻辑一致性。

**4. 辅助任务型角色**
- **帮助智能体 (Help Agent)：** 处理简单的文档问答与操作引导。
- **审批智能体 (Approval Agent)：** 专用于解析人机协同过程中的审批结果，处理用户的确认、修改或拒绝反馈。
- **巡检智能体 (Inspection Agent)：** 以后台定时任务的形式运行，主动扫描集群异常，生成巡检报告。

### 3.3.2 多智能体协作机制设计

**1. MASState与工件（Artifacts）设计**
在多智能体协作中，为了避免信息的隐式丢失与重复传递，Mops框架摒弃了简单的“消息历史追加”模式，转而采用以**多智能体系统状态（MASState）**为核心的共享黑板架构（Blackboard Architecture）。
可以认为，系统中的 Agent 本质上是“有状态的LLM结合感知模块”。MASState 是一个结构化的全局对象，包含：
- `round`：当前协调轮数。
- `plan`：当前任务的执行计划（Plan Artifact）。
- `observation`：探索器收集到的证据快照（Observation Artifact）。
- `action_history`：已执行的工具调用记录。

不同的Agent通过读写特定类型的**工件（Artifacts）**来共享信息。例如，规划器生成 `PlanArtifact` 写入黑板；探索器读取 `PlanArtifact` 的候选工具，调用工具后生成 `ObservationArtifact` 写入黑板。这种基于工件的解耦设计，使得每个Agent的提示词只包含其所需的最小上下文，极大降低了Token开销。

**2. 协调器与阶段状态机**
多智能体的编排模式选择是一个关键问题。Mops选择了“基于协调器的动态状态机编排”而非静态的工作流（Workflow）。协调器根据算法2评估MASState：
- 首轮且无计划时，状态机流转至 `Plan`。
- 如果存在待确认的动作且用户刚刚完成授权，则进入 `Resume` 阶段收束。
- 当写操作参数已准备就绪且观察证据充分时，进入 `Execute`。
这种动态编排允许系统在执行受阻时，灵活地回退到探索或重规划阶段。## 3.4 记忆与上下文管理

智算平台的运维不仅需要关注系统本身，还需要紧密结合当前操作者的语境。为了实现真正智能的交互，Mops框架设计了多源异构数据的统一整合机制。

### 3.4.1 多源异构数据与上下文建模

在真实的运维场景中，数据来源于多个维度，且格式各异。系统必须整合以下几类数据：
1. **用户画像数据（User Profile）：** 包括用户的唯一身份标识、所属组织/租户、资源配额以及历史偏好。这些数据直接决定了用户能否调用高权限的工具。
2. **会话上下文数据（Session Context）：** 记录了当前对话历史与多轮交互的状态。
3. **页面上下文数据（Page Context）：** 当用户在Web前端浏览特定页面（如正在查看JobID为`train-task-1024`的详情页）时，前端会通过隐藏的元数据将这一上下文注入到请求中。
4. **工具返回数据（Tool Observations）：** API查询结果、数据库检索集以及日志内容。
5. **平台元数据（Platform Metadata）：** 全局的资源状态、告警事件流等。

这些异构数据被统一建模并装载到 MASState 中，确保了任何一个被激活的智能体都能获得全局一致的视图。

### 3.4.2 页面、会话与用户上下文的获取与管理

- **用户上下文获取：** 通过前端代理透传的身份Token（如JWT），后端解析并提取用户身份，进一步映射至RBAC权限表。
- **页面上下文注入：** 前端在发送聊天请求时，将当前URL路径与关注的实体ID（Entity ID）封装在Payload的特定字段中。意图路由代理（Intent Router）会优先读取该字段，实现“所见即所问”。
- **会话连贯性维护：** 后端服务通过会话ID（SessionID）索引历史消息。针对大模型的上下文窗口限制，采用了如算法3所示的 **Token预算装载机制（TokenBudgetLoader）**。该机制为不同角色的智能体分配不同的上下文配额，并对长文本的工具返回结果执行首尾截断或智能摘要。

### 3.4.3 多轮状态与确认恢复机制

部分运维任务（如删除作业、重启节点）属于高风险操作，必须引入人的介入（Human-in-the-Loop）。当触发这类操作时，任务无法在单次HTTP请求内完成闭环。
为了解决这一问题，Mops框架引入了**任务快照与会话续接技术（Checkpoint & Resume）**（如算法4所示）。
当执行器准备发起高风险工具调用时：
1. 框架冻结当前的 MASState，将其序列化并持久化到存储中（生成 Checkpoint），并生成一个唯一的 `ConfirmationID`。
2. 协调器暂停当前处理流，通过SSE向前端推送 `confirmation_required` 事件。
3. 用户在前端弹出的确认卡片中进行决策（Approve/Modify/Reject）。
4. 用户的决策请求携带 `ConfirmationID` 返回后端。框架从存储中唤醒对应的快照，反序列化恢复 MASState。
5. 验证用户决策后，执行器接管修改后的参数，继续执行后续流程。

## 3.5 跨智算平台适配的工具体系

工具体系是保证框架可迁移性的关键设计。面对不同智算平台（如Kubernetes原生集群、Slurm调度集群或公有云AI平台）底层接口的差异，Mops采用了经典的适配器模式（Adapter Pattern）对工具层进行了深度解耦。

### 3.5.1 工具声明层与执行层解耦

**工具声明层（Declaration Layer）：**
定义工具的名称、参数JSON Schema描述、返回格式以及风险等级等元信息。这一层不涉及任何具体的网络请求代码，而是向大模型提供标准化的功能描述（Function Calling Schema）。

**工具执行层（Execution Layer）：**
负责将声明层解析出的参数转化为具体平台的API调用逻辑。两者通过统一的接口进行绑定。当需要迁移到新的智算平台时，只需在执行层实现新的平台适配器逻辑（如将 Kubernetes API 替换为 Slurm CLI 命令），而工具声明层以及上层的智能体编排逻辑无需任何修改。

### 3.5.2 工具的分类管理与权限投影

工具根据对系统状态的影响程度被严格分类（只读查询类、状态修改类、高风险操作类）。在实际运行中，并非所有工具都向大模型开放。框架在协调器分配任务前，会执行**角色-工具白名单投影（Role-Tool Projection）**。
系统会根据当前智能体的角色限制（如探索器只能使用低风险工具）、用户的真实权限以及上下文白名单，计算出一个工具子集，大模型只能看到该子集内的工具。这从根本上杜绝了由于大模型幻觉导致的越权调用风险。

## 3.6 安全控制与审计机制

智算平台承载着高价值的数据与计算任务，智能运维框架的引入不能以牺牲安全性为代价。因此，Mops框架构建了严密的安全控制与审计机制。

### 3.6.1 权限过滤与拦截

在工具执行引擎的最前端，设置了硬性的权限校验切面（Aspect）。在工具执行前，系统会提取当前的上下文身份信息，并与目标工具所需的权限要求进行比对。如果普通用户试图让智能体执行“删除其他用户的模型文件”的操作，即使智能体由于幻觉生成了该指令，拦截器也会立刻阻断调用，并向用户返回清晰的越权提示信息。

### 3.6.2 操作确认机制与人机协同

对于中高风险的操作，框架强制实施操作确认机制。在前端UI层面，通过定制的交互卡片（如 `ConfirmActionCard`）清晰展示即将执行的操作类型、目标实体、核心参数及其潜在的爆炸半径。为了辅助用户决策，卡片还会展示类似操作的历史案例。
结合3.4.3节的 Checkpoint 机制，用户可以在确认阶段手动修正大模型提取错误的参数（如将删除资源的数目从10改回1），随后系统执行**确认后恢复与续接**，安全地将任务推进到完成态。

### 3.6.3 审计追踪

为了满足合规要求并支持事后复盘，框架实现了全链路的审计追踪。所有的关键动作——包括用户的原始意图、大模型的中间规划、调用的具体工具及参数、人机协同的确认结果以及最终的执行反馈，均会被格式化为审计日志并持久化存储至数据库。这些数据不仅用于安全追溯，也为后续4.4节的质量评估模块提供了真实的评测数据来源。

## 3.7 本章小结

本章详细阐述了面向智算平台的多智能体协作运维框架（Mops）的设计思想。首先，从实际的智算平台场景出发，对运维任务与数据源进行了建模。其次，提出了包含智算平台适配层、工具与安全声明层、多智能体协作层与上下文层的整体分层架构。在此基础上，深入探讨了协调器、规划器、探索器等角色的设计，以及基于 MASState 的协作状态机。针对安全生产这一核心痛点，详细说明了基于 Checkpoint 的确认恢复机制、跨平台适配的工具体系设计以及全链路的审计与权限过滤方案。这些设计为后续的系统工程实现奠定了坚实的理论基础。
