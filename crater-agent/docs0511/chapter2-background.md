# 第 2 章 相关概念与技术

本章梳理本文工作所涉及的三个基础技术领域：智算平台的关键技术与运维对象（2.1）、大语言模型的提示与工具调用技术（2.2）、以及基于大语言模型的多智能体系统（2.3）。这些技术既是 Mops 设计的输入，也是后续章节论述的术语基础。

## 2.1 智算平台关键技术

### 2.1.1 智算平台架构组成

智算平台（AI Computing Platform）是面向人工智能负载（训练、推理、交互式开发）的算力管理与开发平台，其典型架构可分为四层：**硬件层**（GPU / CPU / 网络 / 存储）、**资源管理层**（Kubernetes + 调度器）、**平台服务层**（作业管理、镜像管理、监控、用户与配额）、**用户界面层**（Portal / Admin）。本文工作所依托的 Crater 智算平台总体架构如图 2-1 所示。

**硬件层**：Crater 集群包含 1 个控制节点和 59 个 GPU 计算节点，含 NVIDIA A100 40GB（7 节点 × 43 张）、V100 32GB（38 节点 × 173 张）与 T4 16GB（2 节点 × 4 张），累计 220 张 GPU，配套 InfiniBand RDMA 高速互联网络与 NFS/Ceph 共享文件存储。

**资源管理层**：使用 Kubernetes 作为容器编排核心，使用 **Volcano** 作为批调度器以支持 Gang Scheduling、双层队列（账户级 Guarantee/Capability + 用户级）、弹性借用与回填等大模型训练所需的调度策略；GPU 设备插件支持 vGPU 切分；网络层包含 Calico、MacVLAN 与 RDMA 三种插件。

**平台服务层**：Crater 共封装了 14 个功能模块（图 2-1 第②层）：AI 作业管理、在线开发环境（JupyterLab/WebIDE）、GPU 资源调度与配额、集群与节点管理、文件存储系统、数据资产管理、镜像与环境管理（Harbor）、监控与可观测性、GPU 智能分析（两阶段 LLM 判定）、用户与权限体系（LDAP/RBAC）、告警与审计、自动化运维与治理（定时巡检/过期资源清理）、系统配置与部署、计费与卡时治理。本文新增的"交互增强 Agent 智能运维"模块位列其中，承担把上述 13 个原有模块的能力组织为统一对话入口的中介职能。

**用户界面层**：Portal（用户工作台）面向研究人员，Admin Console（管理后台）面向平台管理员。Crater 平台支持 7 种作业类型（JupyterLab / WebIDE / PyTorch DDP / TensorFlow PS / 单机 / 自定义批处理等），10 个预置 Grafana 仪表盘，集成 DCGM GPU 指标采集（每张 GPU 暴露约 147 个指标，15 秒采样间隔），通过 LDAP 完成用户认证、通过 SMTP 推送告警邮件。

**典型运行数据**（截至 2026 年 4 月，16 个月连续运行）：注册用户 464 人，累计作业 9.4 万、累计告警 6.8 万、失败作业 8 957 条；A100 利用率持续 100%、V100 利用率 93%；平均失败率 15-25%。失败模式分布：exit code 1（应用错误）54%、exit code 137（OOMKilled）27%、exit code 127（命令缺失）4%、exit code 255（未知）6%；FailedMount 7 248 条、FailedScheduling 3 677 条。这些数据是本文 3.1 节任务建模与第 5 章评测数据集构造的真实输入。

### 2.1.2 平台运维对象

与传统信息系统不同，智算平台的运维对象具有更细的颗粒度与更长的生命周期，可以从纵横两个维度刻画：

**纵向（资源-作业-用户）**：底层资源对象包括节点、GPU、队列、配额、镜像、PVC、网络；中间作业对象包括 AIJob（Crater 自定义 CRD）、Pod、Container、PodGroup（Volcano）、Event、Log、Metric 时序数据；上层用户对象包括账号、角色、个人配额、组织配额、卡时记录。

**横向（提交-调度-执行-终止-审计）**：智算作业从提交开始，依次经历队列等待、Gang 调度、Pod 创建、镜像拉取、容器启动、训练执行、checkpoint 保存、终止退出与审计回收等阶段。每个阶段都可能触发故障：提交阶段的镜像不存在、调度阶段的 FailedScheduling、执行阶段的 OOMKilled / NCCL 通信失败、终止阶段的卷挂载残留等。

基于上述纵横两维，本文将运维任务分为四大类：**诊断**（diagnosis，解释失败原因）、**运维**（ops，调整资源/重启/批量处置）、**查询**（query，状态与指标获取）、**作业提交**（submission，辅助提交与配置）。这一分类与 OpsEval[9] 的 8 任务体系兼容，但更贴近智算训练场景。

## 2.2 大语言模型技术

### 2.2.1 提示词与上下文工程

大语言模型（Large Language Model, LLM）以 Transformer 解码器为骨干，通过自回归生成式建模在大规模语料上预训练得到，能够在零样本或少样本条件下完成翻译、问答、代码生成等任务。现代 LLM（如 GPT-4、Claude 3、Qwen 3 系列）通常具备 8K–1M Token 的上下文窗口与函数调用（function calling）能力。

**提示词工程**（prompt engineering）研究如何通过自然语言指令、示例与格式约束引导 LLM 输出。其中最具影响力的几种范式包括：(1) **思维链**（Chain-of-Thought, CoT），通过"让我们一步步思考"激发显式推理；(2) **少样本学习**（few-shot），通过若干示例对将任务格式传授给模型；(3) **角色扮演**（role prompting），通过"你是一个 X 领域的专家"激发领域知识；(4) **结构化输出**（structured output），通过 JSON Schema 或 XML 约束生成可解析的结构化结果；(5) **自洽性**（self-consistency），通过多次采样并投票得到最稳健的答案。

**上下文工程**（context engineering）则进一步把单次 LLM 调用扩展到长程任务中，关注：(i) 把哪些信息纳入上下文；(ii) 信息以何种顺序/格式组织；(iii) 当上下文窗口不足时如何压缩与遗忘。在多智能体场景中，上下文工程的核心是设计**共享状态**与**视图过滤**机制，使得不同角色只看到与自己职责相关的子集，避免无关信息污染。这也是本文 Mops 设计 MASState + StateView 的理论基础。

### 2.2.2 工具调用与 ReAct 模式

LLM 本身只能生成文本，无法直接读写外部世界；**工具调用**（tool use / function calling）通过将外部 API 抽象为可由 LLM 选择和填参的函数，使 LLM 能够与数据库、搜索引擎、Kubernetes API 等环境交互。OpenAI 在 2023 年首次引入函数调用接口，随后 Anthropic、Google、阿里 Qwen 等模型相继支持。一次工具调用通常包含三个步骤：模型生成调用请求（工具名 + 参数）→ 框架执行工具 → 模型读取工具结果并继续推理。

**ReAct**（Reasoning + Acting，Yao 等[10]）把工具调用嵌入思维链推理中，让 LLM 交替生成"Thought"（推理）、"Action"（工具调用）与"Observation"（环境反馈），直到任务终止。ReAct 的优势在于：(i) 推理与行动相互验证，减轻幻觉；(ii) 通过工具反馈引入新事实，缓解知识陈旧；(iii) 单次工具调用范围有限，错误易于回滚。其代价在于：上下文窗口随轮数线性增长，长链任务容易触及上限；缺乏全局规划，可能出现重复或遗漏；终止条件依赖模型自身判断，存在死循环风险。

**Plan-and-Solve**（PS，Wang 等[11]）针对 Zero-shot-CoT 的"漏步"问题，明确把任务分解为两阶段：先生成计划（"Let's first understand the problem and devise a plan to solve the problem"），再按计划执行。Plan-and-Execute 的智能体变种进一步引入 Planner / Executor 两个角色，Planner 不直接调用工具，Executor 严格按 Plan 执行。其优势在于步骤覆盖更全面；其代价是 Planner 容易产出冗余或不可行的计划。

本文 Mops 综合 ReAct 与 Plan-and-Solve 的优点，引入显式的 Planner 与 Verifier 角色，使规划阶段产生粗粒度任务图，执行阶段以 ReAct 思路调度工具，并由 Verifier 在执行后复核证据充分性，形成"规划-执行-验证"的闭环。

## 2.3 基于大语言模型的多智能体系统

### 2.3.1 基于大语言模型的智能体

**智能体**（agent）是能感知环境、做出决策并产生行动的自治实体。在大语言模型时代，单个智能体通常由三部分组成：(i) 一个 LLM 作为推理引擎；(ii) 一套工具作为执行手段；(iii) 一份提示词作为角色定义。早期代表性工作包括 AutoGPT、BabyAGI、Voyager 等，它们以 ReAct 为骨干循环，通过自然语言反思（reflection）与目标分解（task decomposition）支撑长程任务。

智能体相较纯 LLM 的核心增益在于"行动接口"：通过工具调用，LLM 可以读取实时信息、操作外部系统、写入持久化存储，从被动的"问答模型"变为主动的"执行体"。在运维场景中，智能体的"环境"是 Kubernetes 集群、监控系统、日志数据库等；"工具"则是封装的 kubectl 命令、PromQL 查询、kubectl edit 等接口。

### 2.3.2 基于大语言模型的多智能体

**多智能体系统**（Multi-Agent System, MAS）由多个具有不同角色、能力或目标的智能体组成，它们通过协作（cooperation）、协商（negotiation）或竞争（competition）共同完成复杂任务。把多智能体与 LLM 结合的主要动机包括：

- **职责分工**：单 Agent 难以同时兼顾规划、执行与验证三种风格的推理；分工后可针对每个角色调优提示词和模型。
- **错误隔离与互检**：不同 Agent 对同一证据独立判断，可降低单点幻觉风险。
- **上下文压缩**：每个 Agent 仅需感知自身职责相关的子状态，可显著降低 Token 消耗。
- **可观测性**：多 Agent 的中间步骤天然形成审计轨迹，便于人类理解与回放。

然而多智能体并非天然优于单智能体。从研究角度看，其潜在收益在于更清晰的任务分工、更丰富的中间审查和更好的可解释性；其代价则体现在时延增加、Token 成本上升、角色之间协调不当可能导致重复工作或死循环。Wang 等的实验表明：在简单任务上多 Agent 反而劣于单 Agent；只有在任务步骤数较多、需要多轮证据收集或涉及风险决策时，多 Agent 的优势才显著。这也是本文 Mops 在第 5 章着重在"诊断/运维/提交"这类多轮、有风险的任务上进行评测的原因。

### 2.3.3 多智能体框架

近两年涌现了多种多智能体框架，按编排范式可分为四类：

**(1) 主管-工人模式（Supervisor-Worker）**：以 LangChain 的 LangGraph、CrewAI 为代表。一个 Supervisor Agent 负责调度多个 Worker Agent，Worker 之间不直接通信，通过 Supervisor 中转。优点是控制流清晰、易于调试；缺点是 Supervisor 易成为瓶颈，难以处理高并发任务。

**(2) 对话式模式（Conversational）**：以 AutoGen、MetaGPT 为代表。多个 Agent 通过自然语言对话协作，每个 Agent 可以"发言"或"倾听"。优点是灵活、富表达；缺点是 Token 成本高，对话发散难收敛。

**(3) 状态机模式（State Machine）**：以 STRATUS、KubeIntellect、LangGraph 高级用法为代表。把 Agent 协作建模为有向状态图，节点是 Agent，边是状态转移条件。优点是可控、可形式化分析；缺点是灵活度受限，扩展新 Agent 需要修改图。

**(4) 工件共享模式（Artifact-based）**：把多 Agent 协作的中间产物（计划、观察、动作）建模为结构化工件存于共享状态对象，每个 Agent 读写自己关心的工件字段，无需直接消息通信。优点是状态可序列化、可回放、可与外部系统集成；缺点是工件设计需要前期投入。

本文 Mops 采用 **工件共享 + 状态机** 的混合模式：以 `MASState` 作为共享状态对象承载 Goal/Plan/Observation/Execution 四类工件，Coordinator 作为状态机控制循环走向，避免单纯主管模式下 Supervisor 成为瓶颈，也避免对话模式下的 Token 浪费。

**人在回路（HITL）**机制是多智能体在生产环境的必备组件。在涉及不可逆操作（删除资源、修改配额、重启训练作业）时，系统需要暂停执行、向人类请求确认、得到批准后恢复执行。HITL 的关键技术包括：(i) 风险等级标注，自动判断哪些操作需要确认；(ii) 工作流检查点（checkpoint），把执行状态序列化以便确认期间不持有运行时上下文；(iii) 续接机制，确认完成后从 checkpoint 恢复到原任务流；(iv) 确认 UI 展示与操作内容、潜在影响、历史案例。本文 Mops 把 HITL 作为一等公民，通过 SSE pause 事件与 workflow checkpoint 实现轻量级的人在回路。

## 2.4 本章小结

本章梳理了智算平台关键技术、大语言模型提示与工具调用机制、以及基于 LLM 的多智能体系统三方面基础概念。智算平台的运维对象由资源、作业、用户三层与提交-调度-执行-终止-审计五阶段交织而成，其复杂度显著高于传统 IT 系统；大语言模型通过 ReAct 与 Plan-and-Solve 等范式具备了工具调用与长程推理能力，但单 Agent 在多轮、高风险任务上仍存在步骤遗漏与幻觉问题；多智能体框架通过角色分工与工件共享可缓解上述问题，但需要精心设计协作协议与人在回路机制。下一章将基于上述概念，详细阐述 Mops 多智能体协作框架的设计。
