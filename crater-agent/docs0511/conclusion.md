# 结论

本文针对智算平台运维任务复杂、用户群体多元、操作风险敏感等核心挑战，设计并实现了一种面向智算平台的多智能体运维系统 Mops。论文工作主要可以总结为以下四点：

**(1) 智算平台运维任务建模。** 在分析 Crater 智算平台 16 个月真实运行数据（9.4 万作业、6.8 万告警、8 957 条失败记录）的基础上，把面向智算平台的运维任务抽象为诊断、运维、查询、作业提交四类；把运行上下文形式化为 $C=(\text{actor}, \text{page}, \text{capabilities})$ 三元组；把每个工具形式化为 $T=(\text{name}, \text{schema}, \text{executor}, \text{permission}, \text{confirm})$ 五元组。这一建模为后续多智能体协作与跨平台迁移提供了形式化基础。

**(2) 多智能体协作框架设计。** 提出由协调器、规划器、探索器、执行器、验证器五个核心角色与帮助、审批、配额、巡检四个任务型角色组成的多智能体协作框架；采用"工件共享 + 阶段状态机"的混合协作模式，以 `MASState` 作为跨角色共享的结构化状态对象，通过工件流转完成 Goal→Plan→Observation→Execution→Verify 五阶段闭环；通过 IntentRouter 实现意图分层路由，通过硬上界、无进展守护、工具签名去重三道机制控制长链 Agent 系统的死循环与重复执行风险。

**(3) 智算平台生产级工程实现。** 在 Crater 平台上以"前端 / 后端 / Agent Runtime"三服务架构实现并部署完整系统，通过 SSE 实现流式输出，通过 workflow checkpoint 实现高风险操作确认与续接，通过 PostgreSQL 完成全链路审计；工具层采用声明与执行适配分离的设计，使框架不绑死特定智算平台 API；针对运维操作的安全风险，通过权限过滤、操作确认、确认续接与审计追踪四道闸门保障可控可追溯。

**(4) 评测数据集与系统评估。** 基于 Crater 真实负载构建了包含 85 个典型场景、覆盖四类任务的离线评测集 CraterOps-85，并设计六维评分体系。实验结果表明：在 Qwen3-Coder-Plus 模型下 Mops 整体得分 92.78、工具 F1 0.9889，相较 Plan-and-Solve 提升 4.48 分、相较 Single-ReAct 提升 8.68 分；在涉及多轮确认的运维与提交任务上 Mops 优势达 +10.67 与 +4.02 分，多轮通过率 92.9%；消融实验进一步定位 Explorer 与 Planner 是多 Agent 收益的主要来源。Mops 已在 Crater 平台 Portal 与 Admin Console 灰度上线，承载 1 248 次会话、267 次高风险操作确认，管理员被用户反复问询的工单数量下降 41%。

**研究展望**：本文工作仍存在若干局限与可改进方向。首先，长日志压缩策略偏简单，导致在 7 日级训练日志诊断上偶有遗漏，后续可引入分段语义压缩与关键词反向定位机制；其次，Verifier 在简单任务上偶尔过度保守，可通过引入"通过阈值"或基于难度自适应的提示词进行优化；再次，当前 88 个工具仍以人工声明为主，可借鉴 KubeIntellect 的动态工具合成思想，对自定义 CRD 实现自动适配；最后，本文评测主要在离线快照模式下进行，下一步将构建在线 A/B 实验框架，在生产负载上进一步量化 Mops 的真实价值并扩展到推理服务、镜像安全、数据集治理等场景。
