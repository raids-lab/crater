# 摘要

随着人工智能技术在科研、教学和产业场景中广泛应用，面向机器学习、深度学习和大模型任务的智算平台逐渐成为统一管理算力资源、支撑模型开发和提供用户服务的重要基础设施。此类平台需要同时承载训练、推理、交互式调试、镜像管理、任务调度、资源配额、监控告警和审批审计等多类任务。与传统信息系统运维相比，智算平台的运维对象更加复杂：一方面，计算资源成本较高，作业运行过程依赖集群调度、存储、网络、镜像和运行环境等多层组件；另一方面，平台用户数量和使用频次不断增加，用户在作业提交、状态查询、故障排查、资源选择和环境使用等方面产生了大量自助服务需求。现有运维智能体和评测基准多面向通用云服务、微服务系统或普通集群管理，缺少对智算平台任务语义、工具边界和安全执行流程的系统化支持。

针对上述问题，本文设计并实现了面向智算平台的运维智能体系统 **Mops**（**M**ulti-Agents for Intelligent Computing Platform **Op**eration**s**）。首先，本文对智算平台智能运维任务进行抽象，将用户帮助、状态查询、作业诊断、资源治理、辅助提交、审批评估、巡检报告和受控执行等任务纳入统一能力范围。其次，本文设计了基于角色分工的多智能体协作框架，通过协调、规划、探索、执行和验证等角色完成任务分流、步骤拆解、证据收集、操作执行和结果复核。系统同时引入上下文状态管理机制，将用户身份、页面信息、会话历史、续接状态和工具结果纳入任务处理过程；在工具层采用工具声明与执行适配分离的设计，使智能体框架不依赖某一固定平台接口，便于后续迁移到不同智算平台。针对运维操作中的安全风险，系统设计了权限过滤、操作确认、确认后恢复和审计追踪机制，以保障高风险操作可控、可追溯。

本文基于实际智算平台 Crater 完成了系统的工程落地与生产部署，并构建了面向智算平台的离线评测数据集。该数据集包含 **85 个典型场景**，覆盖诊断、运维、查询和作业提交四类任务，并针对不同场景设计差异化评分规则。在 Qwen3-Coder-Plus 模型上的对比实验显示，Mops 的综合得分为 **92.78**，工具选择 F1 达到 **0.9889**，相较 Plan-and-Solve（88.30）提升 **4.48 分**，相较 Single-ReAct（约 84.10）提升约 **8.7 分**；在运维和提交两类涉及多轮确认的高风险任务上，Mops 相对 Plan-and-Solve 的领先优势分别达到 **10.67 分** 与 **4.02 分**。消融实验进一步表明，规划器（Planner）与验证器（Verifier）协作是 Mops 多智能体优势的关键来源。结果表明，Mops 能够较好地支撑智算平台典型运维任务，在复杂诊断、多轮交互和高风险操作场景中体现出多智能体协作和安全控制机制的有效性。

**关键词**：智算平台；多智能体；运维智能体；大语言模型；人在回路；上下文工程

# Abstract

With the widespread adoption of artificial intelligence in research, education and industry, AI Computing Platforms (ICPs) that schedule heterogeneous GPU resources and serve training, inference and interactive debugging workloads have become critical infrastructure. ICPs must simultaneously support image management, scheduling, quota enforcement, monitoring, alerting and audit-review workflows. Compared to traditional IT systems, ICP operations face a more complex object model — compute is expensive, jobs depend on tightly coupled scheduler / storage / network / image layers, and users (who are typically researchers rather than operators) generate large volumes of self-service requests for diagnosis, resource selection and submission assistance. Existing operations agents and evaluation benchmarks largely target general cloud services or microservices, and lack systematic support for ICP-specific task semantics, tool boundaries and safety-execution flows.

This paper designs and implements **Mops** (**M**ulti-Agents for Intelligent Computing Platform **Op**eration**s**), an operations agent system tailored for AI computing platforms. Mops abstracts user help, status query, job diagnosis, resource governance, submission assistance, approval evaluation, scheduled inspection and controlled execution into a unified capability surface. A role-based multi-agent collaboration framework — coordinator, planner, explorer, executor and verifier — performs intent routing, step decomposition, evidence gathering, action execution and post-hoc validation. A shared `MASState` object integrates user identity, page context, conversation history, resume context and tool results. Tool declaration and tool execution are decoupled via an adapter layer so the framework is portable across heterogeneous ICPs. For safety, Mops introduces permission filtering, operation confirmation, post-confirmation resume and audit logging to keep high-risk operations controllable and traceable.

Mops has been deployed in production on the Crater ICP. An offline benchmark of **85 representative scenarios** spanning four task categories (diagnosis, operations, query, submission) with differentiated scoring profiles has been constructed. On Qwen3-Coder-Plus, Mops achieves an overall score of **92.78** and a tool-selection F1 of **0.9889**, outperforming Plan-and-Solve (88.30) by **+4.48** points and Single-ReAct (~84.10) by about **+8.7** points; on the operations and submission categories that involve multi-turn confirmation, Mops leads Plan-and-Solve by **+10.67** and **+4.02** respectively. Ablations confirm that the planner–verifier collaboration is the primary source of Mops' advantage. The results demonstrate that Mops effectively supports typical ICP operations tasks and that its multi-agent collaboration and safety-control mechanisms are valuable under complex diagnosis, multi-turn dialogue and high-risk-action scenarios.
