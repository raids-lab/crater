# 第 1 章 绪论

## 1.1 研究背景与意义

### 1.1.1 课题来源与背景

本课题来源于国家重点研发计划"先进计算与新兴软件"专项"面向国产异构加速卡集成与优化的智能计算系统"（项目编号 2024YFB4505604）。课题以高校和科研机构常见的云原生智算平台为研究对象，依托北京航天航空大学计算机学院在生产环境中长期运行的智算平台 **Crater** 开展系统设计与工程落地。Crater 由 1 个控制节点和 59 个 GPU 计算节点构成，包含 NVIDIA A100 40GB（7 节点 43 张）、V100 32GB（38 节点 173 张）与 T4 16GB（2 节点 4 张）三类加速卡，**累计 220 张 GPU**，配套 InfiniBand RDMA 网络，注册用户数 **464**；自 2025 年 1 月正式投入运行以来，A100 平均利用率持续保持在 100%，V100 利用率 93%。在统计周期内的 16 个月里，平台共承载 **9.4 万个作业** 与 **6.8 万条告警**，其中超过 56% 的告警与 GPU 低利用率直接相关，平均失败率长期维持在 15%–25% 之间。Crater 是国内高校少数能够连续支撑大规模 AI 训练的开放智算平台之一，其负载特征、用户结构与故障模式具备较强的代表性。

近年来，随着大语言模型与生成式人工智能的快速发展，深度学习已从单卡训练演化为多机多卡分布式训练。OpenAI 的 GPT-4、Anthropic 的 Claude、Google 的 Gemini 以及国内的 Qwen、GLM、DeepSeek 等大型模型动辄需要数千卡同时训练数月。学术界与产业界普遍以"智算平台"这一类专门面向人工智能负载的基础设施承载这些任务。Gartner 在 2024 年的报告中指出，到 2027 年全球 60% 以上的 AI 训练工作负载将在统一的智算平台上完成；中国信息通信研究院《智算平台运维运营技术研究报告》同样指出，智算平台已成为新型基础设施的关键支柱，但其运维模式仍以"人工巡检 + 经验排障"为主，难以适应万卡级规模与大规模团队共享的现实诉求。

智算平台虽然整合了硬件资源，协调统筹分配算力，但是存在易用性较低的问题。用户在提交训练作业、配置镜像和启动命令、查看任务状态、分析失败原因、判断资源等待原因以及进行作业重提时，往往需要同时理解多层异构信息：调度层面的 Pod 与队列状态、容器层面的退出码与日志、监控层面的 GPU 与内存时序数据、平台业务层面的配额与镜像约束、存储层面的挂载状态与数据路径等。此类信息分散于 Kubernetes API、Volcano 调度器、Prometheus 监控、Harbor 镜像仓库、PostgreSQL 业务数据库与文件存储等多个系统中，需要使用者具备较强的工程背景才能整合理解。

平台典型用户并非专职运维人员，而主要是模型训练研究者、研究生和课程实践使用者。该类用户通常熟悉算法、框架和数据处理流程，却未必系统掌握 Kubernetes 调度语义、容器异常退出原因、集群配额机制和资源队列策略。当用户面对"FailedScheduling""OOMKilled""exit code 137""卷挂载失败"等信息时，虽然平台能够展示原始证据，但尚不能自动完成证据整合、语义解释和操作建议。在 16 个月的运行周期内，平台共产生 8 957 条失败作业记录，其中应用错误（exit code 1）54%、内存溢出（exit code 137）27%、命令缺失（exit code 127）4%；同时 7 248 条 FailedMount 与 3 677 条 FailedScheduling 事件被忽视或被多次复发。由此可见，当前平台缺少的并不是原始数据本身，而是对多源证据进行面向用户解释的**交互中介**。

### 1.1.2 课题意义

本课题面向上述挑战，研究并实现一种基于大语言模型多智能体的智算平台运维智能体系统 Mops，使其能够：（1）以自然语言为入口，把碎片化的平台能力组织为可问答、可分析、可确认执行的协作流程；（2）通过角色分工降低单一智能体在长链运维任务中的步骤遗漏与幻觉风险；（3）在涉及写操作时引入权限过滤、操作确认与审计追踪，保证可控、可追溯的"运维大脑"。课题的科学意义在于，把面向云原生服务的智能运维（AIOps）扩展到 GPU 训练作业生命周期管理的新场景，填补当前研究在"智算平台 + 多智能体"交叉领域的空白；工程意义在于，为 Crater 等开放智算平台提供可复用、可演进的运维智能体基础设施，降低用户使用门槛并提升管理员治理效率，对国产智算系统的稳态运营具有直接的参考价值。

## 1.2 国内外研究现状

智算平台运维智能体所涉及的研究脉络可以归纳为三条主线：智算平台与机器学习基础设施的演进、智能运维（AIOps）方法的发展（从传统机器学习到大语言模型驱动的智能体），以及面向云原生与智算平台的多智能体运维系统。下面分四节展开。

### 1.2.1 智算平台相关研究

智算平台以大规模 GPU 集群为算力底座，承担大语言模型预训练、微调与推理服务，其管理面临高异构、强时延敏感和长周期负载的挑战。杜量等[1] 针对万卡级 RDMA 智算集群提出"端网协同"运维框架，包括精细化端网状态监测与告警、面向 AI 集群的"同轨拓扑"自动校验，以及基于改进模拟退火的 AI-DCQCN 自适应拥塞控制，实验显示其方案可将 RDMA 带宽提升约 20% 并显著降低时延。中国信通院《智算平台运维运营技术研究报告》[2] 从基础设施、调度、可观测性、稳定性、运营六个层面系统梳理了智算平台的能力要求，并指出当前业界普遍缺乏"自适应、可解释、可审计"的智能化闭环。曹通[3]、邵明岩[4] 等硕士论文代表了智算平台的"平台侧"研究，将 Kubernetes、容器化、JupyterHub、模型训练与推理服务封装为统一门户，解决"如何让算法工程师便捷使用 GPU"的问题，但未触及"平台自身在万卡规模下如何自我诊断、自我修复"。本课题所依托的 Crater 平台属于此类一站式 AI 平台的代表，但其智能运维侧尚未形成体系化方案。

### 1.2.2 智能运维相关研究

智能运维（AIOps）由 Gartner 于 2017 年提出，涵盖异常检测、日志/指标/调用链分析、事件关联、根因分析与自动化修复。近年来随着大语言模型的发展，AIOps 经历了从"任务专用模型"到"通用智能体"的范式跃迁，可大致划分为三代。

**第一代：传统多模态 AIOps（任务专用、端到端学习）**。Zhang 等[5] 提出的 DeepTraLog 在 ICSE'22 给出 Trace Event Graph 统一表示，使用门控图神经网络配合 Deep SVDD 进行无监督异常检测，在微服务基准上达到 F1 0.95，比基线平均提升 0.37；Lee 等[6] 在 ICSE'23 提出的 Eadro 首次将异常检测与根因定位联合训练，验证了多任务学习在 AIOps 中的价值；Zhao 等[7] 提出的 AnoFusion 使用图 Transformer 与 GAT/GRU 对 metric/log/trace 三模态融合，在两个微服务数据集上分别达到 F1 0.857 与 0.922。这一代方法的共同特征是"训练一个固定模型解一个固定任务"，对数据标注与场景适配性要求较高。

**第二代：LLM 增强的 AIOps（数据增强 + 单步推理）**。Chen 等[8] 提出的 RCACopilot 是首批将 LLM 引入云事件根因分析的工业级系统，已在微软部署 4 年以上，其核心思想是让运维工程师为每类告警预先配置"incident handler"，自动采集多源诊断信息，再交给 LLM 预测根因类别，达到 76.6% 的准确率。Liu 等[9] 在 FSE'25 提出的 OpsEval 构建了首个中英文双语 Ops 基准，覆盖 9 个子域、8 个任务、24 个被评模型，提出 FAE-Score 评测方法，与专家评分相关系数达 0.92。然而 OpsEval 是离线问答评测，无法衡量智能体的"行动正确性"。

**第三代：智能体（Agent）范式**。Yao 等[10] 提出的 ReAct 在 ICLR'23 提出交替生成"推理轨迹（Thought）"与"任务动作（Action）"的范式，使 LLM 既能进行内部推演又能与外部环境交互，在交互式决策任务上以 1–2 个示例就超越模仿/强化学习基线 10%–34%；Wang 等[11] 在 ACL'23 提出 Plan-and-Solve 提示策略，针对 Zero-shot-CoT 的"漏步"问题先"devise a plan"再"carry out the plan"，在 10 个数据集上稳定超过 Zero-shot-CoT。这两种范式是当前几乎所有运维智能体（包括本文 Mops）的"思考-行动"骨干。

近期工作进一步将其推向多智能体协同。Pradhan[12] 提出的多 Agent AIOps 蓝图（监控/诊断/修复/学习/编排五种 Agent）展示了"以 Python 库拼装多智能体"的实用主义视角，但缺乏对安全性、撤销与可审计性的形式化讨论；Raj[13]、Manohar 等[14] 在轻量级商业化落地方面给出了基于 LangGraph、CrewAI 的实现，能将端到端故障恢复 MTTR 从数小时缩短到 28 分钟，但仍未对 LLM 幻觉造成的错误命令做形式化保证。

### 1.2.3 智算平台运维相关研究

这一节聚焦于"针对 Kubernetes 或云原生基础设施的、由多个大模型驱动的智能体协作系统"，与本文工作最为接近。Chen 等[15] 在 NeurIPS'25 提出的 **STRATUS** 是当前最具影响力的自主 SRE 多智能体系统，包含 Detection、Diagnosis、Mitigation、Undo 四个专门化 Agent，以状态机进行确定性编排；其核心理论贡献是 **Transactional No-Regression (TNR)** 安全规范，通过 A-Lock（写者互斥）、Faithful Undo（基于栈式回滚）、Bounded Risk Window（事务长度上界 K=20）三大假设，保证"任意时刻系统外部可见的严重度不会高于初始故障基线"，在 AIOpsLab 与 ITBench 上的失败缓解成功率比 SOTA 提升 1.5 倍以上。

Chen 等[16] 在 MLSys'25 提出的 **AIOpsLab** 是 AgentOps 范式的代表性基准框架：通过 Orchestrator 协调 Agent-Cloud Interface (ACI)、故障注入器、负载生成器与可观测性栈，并基于 DeathStarBench 构造 48 个跨"Detection / Localization / RCA / Mitigation"的运维问题，把 AIOps 问题形式化为 (任务, 上下文, 解决方案) 三元组。Shetty 等[17] 的 SoCC'24 愿景论文进一步提出"自治云"九项设计原则（模块化、灵活接口、可扩展操作尺度等），揭示了现有评测大多是"静态问答 / 单步分类"无法捕捉真实云动态性的关键事实。Ardebili 等[18] 提出的 **KubeIntellect** 以 LangGraph 为编排引擎，按 Kubernetes API 七大类动词划分专门化 Agent，并引入 Code Generator Agent，使系统在 200 条自然语言查询上达到 100% 可靠性与 93% 工具合成成功率，其差异化在于"动态工具合成"。Tang 等[19] 在 2025 CCF AIOps 挑战赛获奖的 MicroRCA-Agent 展示了"LLM Agent + 传统 ML 预处理器"的混合架构在多模态根因定位上的可行性。中文学界，胡金艳等[20] 的 AISci-Friends 提出"知识分析-问题聚焦-实验设计-跨学科运用"四角色协同的多智能体设计范式，为本课题的多 Agent 设计提供了中文参考。

### 1.2.4 现有研究局限性分析

综合 1.2.1–1.2.3 节文献，现有研究在以下五方面存在局限，本文工作正是面向这些不足：

**(1) 场景失配**：STRATUS、AIOpsLab、KubeIntellect、Eadro、AnoFusion 等系统均面向通用云 / 微服务而非 GPU 训练栈。智算平台特有的故障（NCCL 通信失败、checkpoint 异常、镜像污染、跨节点同步、Volcano Gang Scheduling 失败、GPU OOM）在现有基准（DeathStarBench、SocialNetwork、HotelReservation）中并无对应故障模型。**(2) 安全与可恢复性缺失**：多数工作主要演示"能跑通"，未对 LLM 幻觉造成的错误命令做形式化保证；STRATUS 的 TNR 提供了基线，但其 Faithful Undo 假设系统是状态调和式的，对于训练作业中存在的不可逆操作（已删除的 checkpoint、已释放的弹性 GPU 配额）仍需扩展。**(3) 评测缺位**：OpsEval、AIOpsLab、ITBench 都以"问答 + 微服务故障注入"为主，缺乏对"GPU 训练故障"的标准评测集。**(4) 多智能体协同协议薄弱**：现有工作多采用 CrewAI / LangGraph 的"主管 + 工具调用"模型，子 Agent 之间缺乏严格的状态共享、停止条件与递归终止判定。**(5) 工具生态封闭**：KubeIntellect 的合成范围限于 Kubernetes API；MicroRCA-Agent 的工具集需手工预置，缺乏面向智算平台特有 CRD（如 Volcano Queue、Crater AIJob）的扩展机制。

## 1.3 研究目标与研究内容

### 1.3.1 研究目标

本文目标是设计并实现一种**面向智算平台的多智能体运维系统 Mops**，使其在不依赖单一智算平台具体接口的前提下，能够支撑用户与管理员侧的典型自助式运维任务，并通过权限过滤、操作确认、确认后恢复与审计追踪保障高风险操作的安全。具体目标包括：（i）将智算平台运维任务建模为"角色—任务—工具—风险"四元组并形式化抽象；（ii）设计多智能体协作框架，把任务流转、证据收集、操作执行与验证复核分配到不同角色；（iii）实现可在 Crater 平台生产部署的工程系统，支撑从对话交互到执行审计的端到端闭环；（iv）构建针对智算平台的离线评测数据集，量化 Mops 相对于单智能体与单模态提示方法的有效性。

### 1.3.2 研究内容

围绕上述目标，本文开展三方面研究：

**研究内容一：智算平台运行情境感知与工具能力封装。** 把平台已有能力统一抽象为结构化工具，每个工具被形式化为五元组 $T=(\text{name}, \text{schema}, \text{executor}, \text{permission}, \text{confirm})$，并定义只读/写/管理员只读/审批四类风险等级；同时将上下文建模为 $C=(\text{actor}, \text{page}, \text{capabilities})$ 三元组，覆盖用户身份与角色、当前页面位置与关联作业、可用工具集合。

**研究内容二：基于平台任务智能运维的多智能体协作服务搭建。** 设计协调器、规划器、探索器、执行器、验证器五个核心角色，外加帮助、审批、配额、巡检四个任务型 Agent，采用工件共享（MASState）模式而非消息总线模式进行信息传递；引入意图路由、阶段状态机、终止判定与无进展守护，控制单轮 Token 成本与时延。

**研究内容三：面向智算平台智能运维的多智能体系统集成设计。** 在 Crater 平台上以三服务架构（前端 / 后端 / Agent Runtime）实现完整系统，把 Agent Runtime 以镜像化方式部署到 Crater 同一 Kubernetes 集群，通过 Server-Sent Events 完成流式输出，通过 workflow checkpoint 实现确认后恢复，通过 PostgreSQL 完成全链路审计；构建 85 场景离线评测数据集，与 Plan-and-Solve、Single-ReAct 进行系统比较，并通过消融实验定位 Mops 的关键收益来源。

## 1.4 论文组织结构

本文共分为五章：第 1 章为绪论，介绍课题背景、研究现状与研究内容；第 2 章梳理智算平台、大语言模型与多智能体的相关概念与技术；第 3 章给出 Mops 多智能体协作框架的设计，包括任务建模、分层架构、角色定义、上下文管理、跨平台工具体系与安全控制；第 4 章给出 Mops 系统在 Crater 平台上的工程实现，包括三服务架构、数据持久化模型、记忆与上下文实现、多智能体编排与工具/安全模块实现、质量评测模块；第 5 章给出 85 场景离线评测与线上案例分析，包括对比实验、按任务类别与多轮确认流的分析、消融实验和典型案例；最后总结全文工作并展望后续研究方向。
