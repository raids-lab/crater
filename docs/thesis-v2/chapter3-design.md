# 第 3 章 面向智算平台的多智能体协作框架设计

本章是全文的核心。第 2 章已经指出，智算平台运维场景同时承担着"为科研用户解释失败原因"与"为管理员守住资源不可逆边界"两类相互拉扯的责任：前者要求灵活、口语化、能跨越数据源；后者要求严谨、可追溯、对每一次写操作负责。把这两类责任压在单一大语言模型上既不安全也不高效。因此本章提出 **MOps（Multi-agents for Intelligent Computing Platform Operations）** 框架：以多智能体协作为骨架、以共享工件与状态机为协作语言、以零信任工具体系为安全边界，试图在"灵活回答自然语言问题"与"对每一次写操作可追溯"之间找到一个工程上可落地的折中点。

3.1 节抽象智算平台的运维任务、给出框架的设计需求；3.2 节给出 MOps 的设计目标与四层总体架构；3.3 节定义角色与协作机制；3.4 节阐述记忆与上下文管理；3.5 节阐述跨平台工具体系；3.6 节阐述安全与审计机制；3.7 节做本章小结。

## 3.1 智算平台运维任务建模

### 3.1.1 运维对象与用户角色

智算平台的运维对象不像传统 Web 业务那样仅由"服务-数据库-缓存"组成，而是由**资源、作业、用户**三个相互纠缠的子空间构成。资源子空间包括 GPU、CPU、内存、网络拓扑、共享存储与镜像；作业子空间包括交互式 Jupyter、批处理脚本、单机训练、多机分布式训练、推理服务；用户子空间包括科研人员、课题组、平台管理员与外部合作账户。任何一次运维事件都需要从这三个子空间各取一个截面，才能解释清楚——例如"作业失败"既涉及作业自身（提交参数、容器状态），也涉及资源（GPU 健康、节点可调度性），还涉及用户（配额是否被锁定、其他作业是否在抢占）。

为了在工程上承载这种"三维交织"的运维语义，本文把每一次运维请求形式化为一个三元组 $C = (\mathrm{actor}, \mathrm{page}, \mathrm{capabilities})$。$\mathrm{actor}$ 描述请求发起者的身份与组织归属；$\mathrm{page}$ 描述请求发起的页面位置与正在关注的对象；$\mathrm{capabilities}$ 描述当前会话被允许接触的工具集合。三元组在每一轮对话开始时由后端构造完毕、在 LLM 接管之前已经裁剪完毕。这种"先裁后说"的设计思想贯穿全章。

用户角色被刻意分成 **普通用户（user）** 与 **平台管理员（admin）** 两类。两类用户在自然语言入口上没有显式差异，但在工具白名单、确认流强度、巡检权限、审计可见性上完全分层。这种"角色对称-能力非对称"的设计避免了向 LLM 暴露多余的角色字段，也使得后续的零信任过滤（3.5 节）只需要对一个稳定的 $\mathrm{actor.role}$ 字段做投影。

### 3.1.2 任务类型、风险等级与数据依赖

通过对平台一年多真实问询样本的归纳，本文把运维任务划分为四个语义簇：**诊断（Diagnosis）** 解释"为什么 X 出问题"；**运维（Ops）** 完成"对 X 做某种写操作"；**查询（Query）** 直接读取一个统计量；**作业提交（Submission）** 把用户的模糊需求翻译成一份可调度的作业清单。四类任务的核心差异并不在自然语言形式，而是体现在三个隐式属性上：所需数据源的广度、写入风险的等级、与是否必然进入多轮交互。

诊断类任务的数据广度最大、风险最低、多轮可能性中等；运维类任务数据广度中等、风险最高、几乎必然进入多轮（确认）；查询类任务数据极窄、零风险、单轮居多；提交类任务数据中等、中等风险、多轮（参数补全）几乎是必然的。本文据此把每一个工具都额外标注一个**风险等级** $\mathrm{risk} \in \{\mathrm{low}, \mathrm{medium}, \mathrm{high}\}$：仅读为 $\mathrm{low}$；修改用户自有资源为 $\mathrm{medium}$；影响他人或不可逆操作为 $\mathrm{high}$。后两者在执行前必须经过 3.6 节的确认流；前者可直接执行。这一标注是后续所有安全决策的源头。

### 3.1.3 任务建模的数据来源

任务建模并非凭空抽象。本文以平台累计运行 16 个月的真实运行数据为母体，覆盖近九千条失败作业、上万条事件、数万条告警以及九万余次作业提交——这些数据既给出了"运维任务长什么样"的分布形态，也使评测集（第 5 章）能够锚定在真实信号上而非合成 prompt 上。需要强调的是，**建模数据来源** 与 **工具运行时获取数据** 是两件事：建模数据用于设计阶段决定"该框架要解决什么任务、该任务需要哪些工具能力"，而工具运行时数据是这些任务被实际处理时由各类适配器从平台 API 拉取的实时切片。两者通过同一份本体（资源-作业-用户三元）保持语义一致，但生命周期截然不同——前者是设计时一次性的素材，后者是运行时每轮重新装载的证据。

### 3.1.4 设计需求

综上，MOps 框架需要满足以下需求：**FR-1** 以自然语言承接两类用户角色的运维问询，自动绑定页面上下文；**FR-2** 整合至少六类异构数据源；**FR-3** 支持长链多步任务编排；**FR-4** 对中/高风险操作要求显式确认；**FR-5** 支持确认后的状态续接；**FR-6** 支持定时巡检；**FR-7** 全链路审计；**FR-8** 由真实评测驱动迭代。非功能性需求则归纳为六条：**安全可控**、**跨平台可迁移**、**成本可控**、**水平可扩展**、**工程可演进**、**可观测**。这些需求成为 3.2 节设计目标的直接依据。

## 3.2 MOps 框架总体架构设计

### 3.2.1 设计目标

把 3.1.4 节的需求抽象为六条相互掣肘的设计目标：**(D1) 任务覆盖完备**——单一框架同时承接诊断、运维、查询、提交四类任务；**(D2) 多轮交互稳健**——长链对话中能续接、能纠错、能终止；**(D3) 安全可控**——零信任，LLM 输出不可绕过任何权限边界；**(D4) 跨平台可迁移**——核心逻辑不绑死任何具体智算平台；**(D5) 工程可演进**——新增 Agent 或工具不影响既有模块；**(D6) 成本可控**——单轮 LLM 开销与端到端时延都落在可接受预算内。这六条目标既约束本章后续的所有设计选择，也是第 5 章评测指标体系的逻辑来源。

值得注意的是，这六条目标之间存在显著张力：D1 倾向于"加入更多角色与工具"以扩大任务覆盖，但这会与 D6 的成本目标冲突；D3 要求严格收紧权限，但 D2 要求灵活的多轮交互，过度收紧又会让用户体验崩塌。MOps 的设计哲学就是**承认这种张力、用结构化分层而非单点优化来缓解张力**——这正是后面"分层架构 + 工件共享 + 工具声明执行分离 + 零信任"这一连串设计的底层动机。

### 3.2.2 分层架构

如图 3-1 所示，MOps 自顶向下划分为四层：**接入层**承接前端入口与 SSE 事件流；**业务网关层**承担鉴权、工具目录、确认编排、审计落库等"硬约束"职责；**Agent Runtime 层**是本课题的核心，承担意图路由、多智能体编排、记忆管理、工具适配与评测；**基础设施层**是 Kubernetes、Prometheus、PostgreSQL、Harbor、共享存储以及外部 LLM 服务等被运维的对象。

四层架构的关键不在"层数"，而在于两条强约束：第一，**业务网关层是 Runtime 与平台之间唯一的桥**——Runtime 不持有任何 Kubernetes 凭据，所有平台访问都被网关层鉴权并审计；第二，**Runtime 是无状态的**——它持有的所有"记忆"都可以被序列化为快照写入业务网关层的持久化数据库，因此 Runtime 可以被任意水平扩展、任意副本可以承接任意续接请求。这两条约束直接服务于安全（D3）与可扩展（NFR-4）两个目标。

四层之间使用稳定的接口边界相互解耦：接入层与网关层之间是 HTTPS + SSE；网关层与 Runtime 之间是 HTTP + SSE 转发 + 工具回调；网关层与基础设施之间是平台原生 API。这种"接口收敛"的设计让任何一层的替换都不会引起雪崩——例如把 LangGraph 替换为另一种编排引擎、把 Crater 后端替换为另一智算平台后端，理论上都只是替换单层实现。

## 3.3 多智能体角色设计与协作机制

### 3.3.1 智能体角色定义

如果说分层架构回答了"怎么把硬约束从 LLM 边上隔开"，那么角色设计回答的就是"LLM 内部怎么把柔性任务分工到底"。MOps 把多智能体协作的角色分为两组：参与主循环的**核心角色**与独立调度的**任务型角色**，前者形成图 3-2 中以 Coordinator 为中心的协作星图。

**Coordinator（协调器）**是整个 MAS 的中央控制器。它本身不调用工具、不生成最终答复，只负责一件事——判断当前应当进入哪个阶段以及循环是否应当终止。这一限定让 Coordinator 拥有"上帝视角"却不被实际事务打扰，符合管理学中"指挥者与执行者解耦"的经典原则。**Planner（规划器）**根据目标与可用能力生成 Plan 工件，是"该做哪几步"的回答者；它不直接调用工具，否则规划与执行就被混淆，难以单独评测。**Explorer（探索器）**以 Plan 为输入、以只读工具为手段，迭代采集证据并写入 Observation 工件；它对应"看见了什么"。**Executor（执行器）**承接 Plan 与 Observation，对读类问题输出最终答复，对写类问题生成 Action 列表，对应"打算做什么"。**Verifier（验证器）**对 Execution 与 Observation 进行复核，输出 `pass / missing_evidence / risk / retry` 四类裁决；后三类不是简单"打回重做"，而是显式分派回退入口——这是 MOps 区别于线性 Plan-Execute 编排的关键差异。

任务型角色包括 **Guide（帮助）**、**Approval（审批）**、**Quota（配额分析）** 与 **Inspection（巡检）**。它们不参与主 MAS 主循环，而是被独立调度——Guide 用于纯文档问答以避免重型 MAS 启动成本；Approval 是管理员侧的审批专用 Agent，仅对一组白名单工具响应；Quota 用于多账户配额合规性分析；Inspection 由 cron 触发，对全平台失败作业、低利用率作业、长时空闲资源批量分析。这种"主循环 + 旁路 Agent"的双层布局，使得高频简单任务的开销不会被复杂 MAS 拖累、批处理任务也不会与交互对话竞争资源。

### 3.3.2 多智能体协作机制设计

MOps 的协作不依赖角色之间互相"说话"，而是依赖一个名为 MASState 的共享状态对象。每个角色在每一轮被调用时都拿到 MASState 中按角色裁剪后的视图（StateView），把自己的产物写回对应字段，再由 Coordinator 决定下一步。这是一种"工件中心"而非"消息中心"的协作模式：角色之间不需要复述上下文，状态对象本身就是上下文。在图 3-2 中可以清楚地看到，四个核心角色围绕 Coordinator 与四类工件（Goal/Plan/Observation/Execution）形成闭环。

**意图路由与编排模式选择**是协作的前置环节。在进入 MAS 主循环之前，IntentRouter 通过启发式 + 轻量 LLM 的两级策略对用户请求进行粗分类：纯文档帮助、单步查询、长链诊断/运维。两级策略的置信度合并由式 (3.1) 给出：

$$
c_{\text{router}} =
\begin{cases}
  c_{\text{det}}, & c_{\text{det}} \geq 0.75 \\
  \lambda\, c_{\text{det}} + (1-\lambda)\, c_{\text{llm}}, & 0.5 \leq c_{\text{det}} < 0.75 \\
  c_{\text{llm}}, & c_{\text{det}} < 0.5\ \text{且 LLM 输出合法} \\
  0.3, & \text{兜底}
\end{cases}
\tag{3.1}
$$

意图路由的输出决定了 Coordinator 启动哪一档编排模式：纯文档帮助交给 Guide；简单查询走轻量 ReAct；长链任务进入完整 MAS。三档分层的目的是用最小代价覆盖最大任务空间——简单问题不为大循环买单，复杂问题不被简化策略削弱。

**Coordinator 阶段状态机**如图 3-3 所示，由 `INIT→ROUTE→PLAN→EXPLORE→EXECUTE→(PAUSE→RESUME)→VERIFY→SUMMARIZE→END` 九个阶段构成。状态机有两类边：一类是正向推进的实线箭头，由当前工件是否完整决定；另一类是回退分派的虚线箭头，由 Verifier 的裁决驱动。回退分派的逻辑由算法 3.1 给出（基于已有的 LaTeX 实现，下文同）：

```latex
\begin{algorithm}
\caption{协调器阶段决策与回退分派}
\KwData{共享状态 $M$，护栏配置 $\Theta$，路由结果 $\mathcal{R}$，验证器裁决 $v$}
\KwResult{下一阶段 $\phi$}
\uIf{$M.round = 1$ \textbf{and} $M.plan = \varnothing$}{\Return $\text{plan}$}
\uIf{$v.\mathrm{verdict} = \texttt{missing\_evidence}$ \textbf{and} 有未决问题}{\Return $\text{explore}$ \tcp*{① 回退}}
\uIf{$v.\mathrm{verdict} = \texttt{risk}$}{清空 plan 与 actions; \Return $\text{plan}$ \tcp*{② replan}}
\uIf{$v.\mathrm{verdict} = \texttt{retry}$ \textbf{and} 错误可恢复}{对错误动作 $\text{pending}$; \Return $\text{execute}$ \tcp*{③ retry}}
\uIf{同一角色连续两轮无进展}{切换模型/降级模式; \Return $\phi'$ \tcp*{④ reassign}}
\uIf{$M.round \geq \Theta.\mathrm{lead\_max\_rounds}$ \textbf{or} $M.\mathrm{no\_progress} \geq \kappa$}{\Return $\text{finalize}$}
\Return $\textsc{LLMDecide}(M.view, \Theta)$
\end{algorithm}
```

四种回退入口——补充探索、replan、retry、reassign——共同构成 MOps 的"自纠错能力"。线性 Plan-Execute 编排只能"做完一遍交差"，而 MOps 允许"做了一遍发现不对再做一遍"，这是它在复杂诊断任务上能够保持高准确率的关键。第 5 章消融实验也将印证：取消 Verifier 会让简单任务变快但让复杂诊断任务的根因命中率显著下降。

**终止与无进展守护**是长链 Agent 系统的两大隐患。MOps 通过三道守护：硬上界 $\mathrm{lead\_max\_rounds}$、连续无进展计数 $\mathrm{no\_progress\_count}$、工具签名去重 $\mathrm{attempted\_tool\_signatures}$。前者防止死循环；中者防止"重复绕圈但每轮内容一致"的状态停滞；后者防止 LLM 对同一工具用同一参数反复重试。三道守护合在一起，给整个状态机加了一个总时间上界，使得线上稳态运维成为可能。

## 3.4 记忆与上下文管理

### 3.4.1 多源异构数据与上下文建模

MOps 需要整合的数据来源至少包括六类：用户画像（身份/角色/权限/历史偏好）、会话上下文（对话历史/多轮交互状态）、页面上下文（用户当前所看的对象）、任务状态（进行中任务的中间状态与依赖）、工具返回（来自平台 API 的实时切片）以及平台元数据（作业/资源/告警的静态信息）。这六类数据来源在原始形态上千差万别——有的是 JWT 字段、有的是 PostgreSQL 行、有的是 Prometheus 时序、有的是 Kubernetes 事件。要让多个 LLM 角色协同工作，必须把它们统一到一个语义一致的结构化对象上。

本文采用一个名为 **MASState** 的状态对象承担这一统一建模职责（图 3-4）。它把六类异构数据归并为五组字段：四阶段工件、运行控制、证据与审计、确认与恢复、使用统计。在工程上，MASState 是一个 dataclass；在语义上，它是"一次多智能体协作的全部状态"。如果把多智能体协作看作一台有限状态机，那么 MASState 就是这台状态机的全局变量，而每一个角色都是只读特定字段、只写特定字段的纯函数——这种"以状态为枢纽"的设计让角色之间天然解耦。

### 3.4.2 MASState：多智能体共享状态与 StateView

但 MASState 不会被原封不动地交给每个 Agent——这既不必要（每个 Agent 只关心自己的局部）也不安全（避免跨角色串扰）也不经济（token 占用高）。本文为 MASState 引入了一个名为 **StateView** 的投影机制：每个 Agent 在被调用时只看到一个按角色裁剪后的视图。形式化地，StateView 是一个函数 $V_r:\ \mathrm{MASState} \to \mathrm{ViewType}_r$，其中 $r$ 是角色名。Planner 的视图只包含 Goal 与可用能力；Explorer 的视图增加 Plan 与已有 Observation；Executor 视图再增加 actions 草稿；Verifier 视图涵盖 Plan、Observation 与 Execution；Coordinator 视图覆盖全部字段，但以轻量摘要形态出现。

StateView 同时承担 token 预算分配的职责。MOps 按角色分配 token 预算 $B_{\text{role}}$，满足

$$
\sum_{r \in \{P, E, X, V\}} \alpha_r = 1, \quad B_{\text{role}} = \alpha_{\text{role}} \cdot B_{\text{total}}
\tag{3.2}
$$

其中 Planner $\alpha_P \approx 0.20$、Explorer $\alpha_E \approx 0.45$、Executor $\alpha_X \approx 0.25$、Verifier $\alpha_V \approx 0.10$。这一分配遵循"信息进入决策瓶颈处再压缩"的原则：探索阶段是证据最丰富的环节，给最大预算；规划与验证只对结构性信息敏感，给较小预算。在视图装填过程中，若某条工具结果体量超出该角色预算，先尝试结构化截断（保留头尾、剔除中间）、再尝试 LLM 摘要、最后才整体截断。这是"信息不丢失先于 token 不溢出"的实现策略。

把 Agent 抽象为"有状态的 LLM + 感知"是另一种理解：LLM 是无状态推理器，感知则来自 StateView（结构化记忆）与工具返回（实时观察），二者合起来才构成一个具有持续认知能力的 Agent。这一视角解释了为什么 MOps 并不依赖于 LLM 自身的"长上下文窗口"——它把外部状态显式化、可控化，因而即使在中等上下文容量的模型上也能稳定工作。

### 3.4.3 页面、会话与用户上下文

把上下文按生命周期划分为三层（图 3-5）是 MOps 处理多源异构数据的另一关键设计。

**用户上下文（User Context）**是最稳定、最高权威的一层。它来自后端 JWT 鉴权，包含用户标识、账户标识、角色、可见能力域，作为整轮对话的可信源；任何来自前端字段或 LLM 输出的角色信息都不可覆盖此层。这一约束是 3.5 节零信任过滤的事实前提——不允许 LLM 自称管理员是防御提示词注入的最低红线。

**页面上下文（Page Context）**是逐请求更新的隐式参数。前端在用户切换页面时把 route、目标对象、入口动作、时区语言上报至后端。最直接的语义是：当用户在"作业 jobX 的详情页"问"为什么失败"时，Agent 无需用户重复说出作业名，因为页面已经告诉它"jobX 就是焦点对象"。这种以人为本的对话体验依赖前后端协同：前端忠实上报，后端忠实注入，Runtime 忠实使用。

**会话上下文（Session Context）**是跨轮持久化的对话历史。Runtime 在每次请求时只从后端取最近若干轮的历史，并按 token 预算装载——历史消息按时间逆序加入直至预算耗尽；超长的工具结果在历史中以"头部+尾部+省略号"截断，错误结果保留更多文本以便诊断；非结构化文本则交由轻量 LLM 摘要。整个装载逻辑（算法 3.2）以 $B$ 为总预算、以 $b_t$ 为单工具预算，本质上是一个贪心装箱问题。

```latex
\begin{algorithm}
\caption{Token 预算装载}
\KwData{消息历史 $H$（时间逆序），工具结果 $T$，总预算 $B$，单工具预算 $b_t$}
\KwResult{装载后的上下文 $\mathcal{C}$}
$\mathcal{C} \gets \varnothing$;\ $u \gets 0$\;
\ForEach{$h_i \in H$}{
  \If{$u + \mathrm{tok}(h_i) > B$}{\textbf{break}}
  $\mathcal{C} \gets \mathcal{C} \cup \{h_i\}$;\ $u \mathrel{+}= \mathrm{tok}(h_i)$\;
}
\ForEach{$t \in T$}{
  $r' \gets$ \uIf{$\mathrm{tok}(t.r) \leq b_t$}{$t.r$}
   \uElseIf{结构化}{$\textsc{HeadTailTruncate}(t.r, b_t)$}
   \Else{$\textsc{LLMSummarize}(t.r, b_t)$}\;
  $\mathcal{C} \gets \mathcal{C} \cup \{(t, r')\}$\;
}
\Return $\mathcal{C}$
\end{algorithm}
```

### 3.4.4 多轮状态与确认恢复机制

当任务进入"必须等待人类干预"的状态时，MOps 采用 **workflow checkpoint** 机制实现状态续接。其哲学是：与其让 Runtime 在等待期间维持长连接、持有 LLM 上下文（既贵又脆），不如把当前 MASState 序列化为一份快照，连同待确认动作一起交给业务网关层持久化，等用户作出决定后再让任意 Runtime 副本反序列化、继续推进。

这一机制使 Runtime 在确认期间真正成为"无状态服务"——它不持有任何与会话有关的内存状态，等同于一个普通的请求/响应服务。这是 D2（多轮交互稳健）与 NFR-4（水平可扩展）能够同时成立的工程实现基础。

## 3.5 跨智算平台适配的工具体系

**工具声明层与工具执行层的分离**是 MOps 跨平台可迁移性的核心。每个工具被形式化为一个五元组 $T = (\mathrm{name}, \mathrm{schema}, \mathrm{executor}, \mathrm{permission}, \mathrm{confirm})$，其中前四项与具体平台无关，仅 $\mathrm{executor}$ 指向一个抽象适配器接口（图 3-6）。生产环境下使用面向 Crater 后端的适配器，开发环境使用本地适配器，评测环境使用回放快照的 Mock 适配器；迁移到新智算平台时，只需要实现一个新的适配器子类，工具声明、提示词、协作逻辑都不需要改动。这种"声明 vs 执行"的分离对应面向对象设计中的"接口隔离原则"：把 *what* 与 *how* 解耦。

工具按权限语义分为只读、用户写、管理员写、巡检专用四大类，并形成一份**权限矩阵** $\Pi(r, p)$——给定用户角色 $r$ 与页面上下文 $p$，矩阵给出可见工具集合：

$$
\Pi(r, p) = \{ t \in \mathcal{T} \mid \mathbb{1}[t.\mathrm{risk} \in \mathcal{L}_r] \cdot \mathbb{1}[\mathrm{admin}(t) \Rightarrow \mathrm{admin}(r)] \cdot \mathbb{1}[t \in \mathcal{W}_p] = 1 \}
\tag{3.3}
$$

三个示性函数分别表达**角色风险约束**、**用户权限约束**与**页面白名单约束**。该投影在每一轮对话的最初被 SanitizeTools 算法计算一次（算法 3.3），结果作为工具白名单交给 LLM。这一过滤的关键不在算法本身，而在它发生在 LLM 看到工具描述**之前**——LLM 无法"请求"它根本看不见的工具，提示词注入攻击因此被釜底抽薪。

```latex
\begin{algorithm}
\caption{零信任工具白名单过滤 SanitizeTools}
\KwData{上下文 $C$，已启用工具 $\mathcal{T}_e$}
\KwResult{对该会话可见的工具子集 $\mathcal{V}$}
$r^* \gets C.\mathrm{actor.role}$ \tcp*{原则1：角色仅来自 Backend JWT}
\If{$C.\mathrm{page.is\_user\_page}$ \textbf{and} $r^* = \mathrm{admin}$}{$r^* \gets \mathrm{user}$ \tcp*{原则2：可收窄不可放大}}
$\mathcal{V} \gets \varnothing$\;
\ForEach{$t \in \mathcal{T}_e$}{
  \If{\textsc{IsAllowed}($r^*, t.\mathrm{permission}, t.\mathrm{risk}$)}{$\mathcal{V} \gets \mathcal{V} \cup \{t\}$ \tcp*{原则3：LLM 看见之前裁剪}}
}
\Return $\mathcal{V}$
\end{algorithm}
```

## 3.6 安全控制与审计机制

智算平台运维操作的不可逆性——已删除的 checkpoint、已释放的卡时、已驱逐的节点——决定了安全是 MOps 的一等公民。MOps 把安全约束设计为四道串联的闸门（图 3-7），每一道闸门负责一个独立的失败模态。

**闸门 ① 权限过滤**在 LLM 注入之前完成（算法 3.3）。其立意是"治未病"：与其在 LLM 输出阶段事后拦截，不如让无权访问的工具根本不进入 LLM 视野。这一闸门同时承担"页面 scope 不可放大"原则——管理员在浏览普通用户页面时被自动降级为普通用户视角，避免因角色错位而误触高风险工具。

**闸门 ② 操作确认**针对 medium/high 风险工具。系统在执行前向用户推送确认卡片，清晰展示操作类型、目标对象、潜在影响（"将会停止 jobX 正在进行的 3000 步训练"）与历史相似案例。批量操作支持逐项勾选，避免"全选执行"的认知陷阱。确认机制本质上是 **HITL（Human-in-the-Loop）** 的工程化——MOps 不试图替代人类判断，而是把人类判断引入到关键节点。

**闸门 ③ 确认续接**依赖 3.4.4 节的 workflow checkpoint。当用户做出决定后，系统从快照恢复状态、把确认结果合并到 actions 列表、继续执行。整个续接过程对 Runtime 是无状态的——任何副本都能拾起任何中断的会话。续接算法骨架如下：

```latex
\begin{algorithm}
\caption{Pause-Confirm-Resume 续接}
\Function{Pause}{$M, A_{\mathrm{pending}}$}{
  $\sigma \gets \textsc{Serialize}(M, \mathrm{ver}=5)$\;
  $\textsc{Emit}(\texttt{pause}, \{A_{\mathrm{pending}}, \sigma\})$;\ $\textsc{Persist}(\sigma)$\;
}
\Function{Resume}{$\mathrm{req}$}{
  $M \gets \textsc{Deserialize}(\mathrm{req.continuation.ckpt})$\;
  \ForEach{$d \in \mathrm{req.confirmation\_results}$}{
    $a \gets \textsc{Find}(M, d.\mathrm{confirm\_id})$;\ $a.\mathrm{status} \gets d.\mathrm{decision}$\;
  }
  \Return \Call{MasLoop}{$M$}\;
}
\end{algorithm}
```

**闸门 ④ 审计追踪**对所有运维操作（包括被拒绝的尝试）做全链路记录，含发起者、工具名、入参、结果或错误、延时、风险等级、确认 ID 等字段。除事件级日志外，MOps 同时把每轮的 tool_records 与 action_history 整体序列化保留，使得任何一次运维事件都可以"完整回放"——这一性质对事后归因与合规审查至关重要。

四道闸门联合保障一个朴素的安全不变量：**任意时刻，系统在用户视角下能够看到的风险等级，不会高于初始基线**。这一不变量由闸门 ① 与 ② 联合给出实时保障，由闸门 ④ 提供事后核查。MOps 借鉴 TNR（Transactional No-Regression）思想并将其延伸到智算训练作业的不可逆操作，使得多智能体系统在保留 LLM 灵活性的同时获得接近传统运维平台的安全等级。

## 3.7 本章小结

本章给出了 MOps 框架的系统设计。围绕 3.1 节抽象的四类运维任务与 8+6 条需求，3.2 节给出四层总体架构与六条设计目标，3.3 节定义五个核心角色 + 四个任务型角色并通过 Coordinator 状态机与四类回退分派构造多智能体协作机制，3.4 节通过 MASState 与 StateView 把六类异构数据收敛为统一的结构化共享状态并以 token 预算实现按角色裁剪，3.5 节通过工具声明-执行分离与权限矩阵保障跨平台可迁移与零信任，3.6 节通过四道安全闸门与 Pause-Confirm-Resume 续接机制保障安全不变量。本章的设计将在第 4 章被工程化为运行在 Crater 平台上的具体实现。
