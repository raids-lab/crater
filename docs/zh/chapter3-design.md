# 第 3 章 面向智算平台的多智能体协作框架设计

第 2 章已经指出，智算平台的运维场景同时承担"为科研用户解释失败原因"与"为管理员守住资源不可逆边界"两类相互拉扯的责任：前者要求灵活、口语化、能跨数据源整合证据；后者要求严谨、可追溯、对每一次写操作负责。把两类责任压在单一大语言模型上既不安全也不经济。为此，本章提出面向智算平台的多智能体协作框架 **MOps（Multi-agents for Intelligent Computing Platform Operations）**：以角色化的多智能体协作为骨架，以共享工件与状态机为协作语言，以零信任工具体系与人机协同闸门为安全边界，在"灵活回答自然语言问题"与"对每一次写操作可追溯"之间找到一个工程上可落地的折中点。本章 3.1 节对运维任务建模与设计需求做形式化抽象；3.2 节给出总体架构；3.3 节定义角色与协作机制；3.4 节阐述记忆与上下文管理；3.5 节阐述跨平台工具体系；3.6 节阐述安全控制与审计机制；3.7 节小结。

## 3.1 智算平台运维任务建模

### 3.1.1 运维对象与用户角色

智算平台的运维对象并非传统 Web 业务的"服务—数据库—缓存"三段式结构，而是由**资源、作业、用户**三个相互交织的子空间共同构成。资源子空间覆盖 GPU/CPU 节点、共享存储、网络拓扑与容器镜像；作业子空间覆盖交互式 Jupyter、批处理脚本、单机训练、多机分布式训练以及推理服务；用户子空间覆盖科研人员、课题组、平台管理员与合作账户。任何一次运维事件都需要从三个子空间各取一个截面才能解释清楚——例如"作业失败"既涉及作业自身（提交参数、容器状态），也涉及资源（GPU 健康、节点可调度性），还涉及用户（配额是否锁定、其他作业是否抢占）。

为了在工程上承载这种"三维交织"的运维语义，本文把每一次运维请求形式化为一个上下文三元组：

$$C = (\mathrm{actor},\ \mathrm{page},\ \mathrm{capabilities})$$

其中 $\mathrm{actor}=(\mathrm{user\_id},\mathrm{role},\mathrm{account\_id})$ 描述请求发起者的身份与组织归属；$\mathrm{page}=(\mathrm{route},\mathrm{job},\mathrm{node},\mathrm{pvc})$ 描述请求发起的页面位置与正在关注的对象；$\mathrm{capabilities}=\{T_i\}$ 描述当前会话被允许接触的工具集合。三元组在每一轮对话开始时由后端构造完毕、在 LLM 接管之前已经裁剪完毕，"先裁后说"的设计思想贯穿全章。

用户角色被刻意划分为**普通用户（user）**与**平台管理员（admin）**两类。两类用户在自然语言入口上没有显式差异，但在工具白名单、确认流强度、巡检权限、审计可见性上完全分层。这种"角色对称、能力非对称"的设计避免向 LLM 暴露多余的角色字段，也使得后续的零信任过滤只需要对一个稳定的 $\mathrm{actor.role}$ 字段做投影。

### 3.1.2 任务类型、风险等级与数据依赖

对平台一年多真实问询样本的归纳显示，智算运维任务可以划分为四个语义簇：**诊断（Diagnosis）**解释"为什么 X 出问题"，**运维（Ops）**完成"对 X 做某种写操作"，**查询（Query）**直接读取一个统计量，**作业提交（Submission）**把用户的模糊需求翻译成可调度的作业清单。四类任务的核心差异并不在自然语言表面，而是体现在三个隐式属性上：所需数据源的广度、写入风险的等级，以及是否必然进入多轮交互。诊断类任务数据广度最大、风险最低、多轮可能性中等；运维类任务数据广度中等、风险最高、几乎必然进入多轮确认；查询类任务数据极窄、零风险、单轮居多；提交类任务数据中等、风险中等、多轮参数补全几乎是必然。

在此分类之上，本文为每一个工具额外标注一个**风险等级** $\mathrm{risk}\in\{\mathrm{low},\mathrm{medium},\mathrm{high}\}$：仅读为 low，修改用户自有资源为 medium，影响他人或不可逆操作为 high。后两者在执行前必须经过 3.6 节的确认流，前者可直接执行。这一字段不仅是工具调用的标签，更是后续所有安全决策的源头——多智能体编排、确认机制、审计回放都以风险等级为枢轴。

### 3.1.3 任务建模的数据来源

任务建模并非凭空抽象，而是以平台累计运行 16 个月的真实运行数据为母体——覆盖近九千条失败作业、上万条调度与挂载事件、数万条告警以及九万余次作业提交。这些数据既给出"运维任务长什么样"的分布形态，也使后续评测集能够锚定在真实信号上而非合成 prompt 上。

需要强调的是，**建模数据来源**与**工具运行时获取数据**是两件事：前者用于设计阶段决定"框架要解决什么任务、任务需要哪些工具能力"，是设计时一次性的素材；后者是任务被实际处理时由各类适配器从平台 API 拉取的实时切片，每轮重新装载。两者通过同一套"资源—作业—用户"本体保持语义一致，但生命周期截然不同——这一区分直接对应 3.5 节的工具声明层与执行层分离。

综上，MOps 框架需要同时满足若干功能性需求与若干非功能性需求。功能性需求包括：以自然语言承接两类用户角色的运维问询并自动绑定页面上下文；整合至少六类异构数据源；支持长链多步任务编排；对中/高风险操作要求显式确认；支持确认后的状态续接；支持定时巡检；提供全链路审计；可由真实评测数据驱动迭代。非功能性需求归纳为六条：安全可控、跨平台可迁移、成本可控、水平可扩展、工程可演进、可观测。这些需求成为下一节设计目标的直接依据。

## 3.2 MOps 框架总体架构设计

### 3.2.1 设计目标

把上一节的需求抽象为六条互相掣肘的设计目标：**(D1) 任务覆盖完备**——单一框架同时承接诊断、运维、查询、提交四类任务；**(D2) 多轮交互稳健**——长链对话能续接、能纠错、能终止；**(D3) 安全可控**——零信任，LLM 输出不可绕过任何权限边界；**(D4) 跨平台可迁移**——核心逻辑不绑死任何具体智算平台；**(D5) 工程可演进**——新增智能体或工具不影响既有模块；**(D6) 成本可控**——单轮 LLM 开销与端到端时延都落在可接受预算内。这六条目标既约束本章后续的所有设计选择，也是评测指标体系的逻辑来源。

六条目标之间存在显著张力。D1 倾向于"加入更多角色与工具"以扩大任务覆盖，但这会与 D6 的成本目标冲突；D3 要求严格收紧权限，但 D2 要求灵活的多轮交互，过度收紧又会让用户体验崩塌。MOps 的设计哲学并非寻找单点最优，而是**承认这种张力、用结构化分层而非单点优化来缓解张力**。这正是后续"分层架构 + 工件共享 + 工具声明执行分离 + 零信任过滤"这一连串设计的底层动机。

### 3.2.2 分层架构

如图 3-1 所示，MOps 自顶向下划分为四层。**接入层**承接前端入口与 SSE 事件流，是用户感知系统的窗口；**业务网关层**承担鉴权、工具目录维护、确认编排、审计落库等"硬约束"职责，是系统的安全边界；**Agent Runtime 层**是本课题的核心，承担意图路由、多智能体编排、记忆管理、工具适配与评测；**基础设施层**是 Kubernetes、Prometheus、PostgreSQL、Harbor、共享存储以及外部 LLM 服务等被运维的对象。

四层架构的关键不在"层数本身"，而在于两条强约束。第一，**业务网关层是 Runtime 与平台之间唯一的桥**——Runtime 不持有任何 Kubernetes 凭据，所有平台访问都被网关鉴权并审计。这一约束直接服务于安全目标 D3：即便 Runtime 被攻陷，攻击者拿到的也不过是一个无凭据的 LLM 推理服务。第二，**Runtime 是无状态的**——它持有的所有"记忆"都可以被序列化为快照写入网关层的持久化数据库，因此 Runtime 可以任意水平扩展，任意副本可以承接任意续接请求。这两条约束直接服务于安全（D3）与可扩展两个目标。

四层之间使用稳定的接口边界相互解耦：接入层与网关层之间是 HTTPS + SSE，网关层与 Runtime 之间是 HTTP + SSE 转发 + 工具回调，网关层与基础设施之间是平台原生 API。这种"接口收敛"的设计使任何一层的替换都不会引起雪崩——把 LangGraph 替换为另一种编排引擎、把 Crater 后端替换为另一个智算平台后端，理论上都只是替换单层实现。

## 3.3 多智能体角色设计与协作机制

### 3.3.1 智能体角色定义

如果说分层架构回答了"怎么把硬约束从 LLM 边上隔开"，那么角色设计回答的就是"LLM 内部怎么把柔性任务分工到底"。MOps 把多智能体协作的角色分为两组：参与主循环的**核心角色**与独立调度的**任务型角色**。

**意图路由代理（IntentRouter）**作为整个系统的流量入口，对用户的自然语言输入、页面上下文以及历史会话进行粗分类。它采用启发式规则与轻量级 LLM 结合的两级策略：当检测到特定页面上下文（如作业详情页）时直接生成高置信意图，进入对应子工作流；对于复杂语义则调用 LLM 进行意图槽位提取，最终决定走纯文档帮助、单步 ReAct 还是完整 MAS 主循环。三档分层的目的是用最小代价覆盖最大任务空间——简单问题不为大循环买单，复杂问题不被简化策略削弱。

**协调代理（Coordinator）**是整个 MAS 的中央控制器。它本身不调用工具、不生成最终答复，只负责一件事——判断当前应当进入哪个阶段以及循环是否应当终止。这一限定让 Coordinator 拥有"上帝视角"却不被实际事务打扰，符合管理学中"指挥者与执行者解耦"的经典原则。**规划器（Planner）**根据目标与可用能力生成 Plan 工件，是"该做哪几步"的回答者；它不直接调用工具，否则规划与执行被混淆，难以单独评测与迭代。**探索器（Explorer）**以 Plan 为输入、以只读工具为手段，迭代采集证据并写入 Observation 工件；它对应"看见了什么"。**执行器（Executor）**承接 Plan 与 Observation，对读类问题输出最终答复，对写类问题生成 Action 列表，对应"打算做什么"。**验证器（Verifier）**对 Execution 与 Observation 进行复核，输出 $\texttt{pass}\,/\,\texttt{missing\_evidence}\,/\,\texttt{risk}\,/\,\texttt{retry}$ 四类裁决，后三类不是简单"打回重做"，而是显式分派回退入口——这是 MOps 区别于线性 Plan-Execute 编排的关键差异。

任务型角色独立于主循环：**帮助代理（Guide）**处理简单文档问答与操作引导，避免为轻任务启动重型 MAS；**审批代理（Approval）**专用于解析人机协同过程中的审批结果，仅对一组白名单工具响应；**配额代理（Quota）**用于多账户配额合规性分析；**巡检代理（Inspection）**以 cron 触发，对失败作业、低利用率作业、长时空闲资源做批量分析并生成报告。"主循环 + 旁路 Agent"的双层布局使得高频简单任务不被复杂 MAS 拖累，批处理任务也不会与交互对话竞争资源。

### 3.3.2 多智能体协作机制设计

MOps 的协作不依赖角色之间互相"说话"，而是依赖一个名为 **MASState** 的共享状态对象。每个角色在每一轮被调用时拿到 MASState 中按角色裁剪后的视图，把自己的产物写回对应字段，再由 Coordinator 决定下一步。这是一种"工件中心"而非"消息中心"的协作模式：角色之间不需要复述上下文，状态对象本身就是上下文。MASState 中显式持有四类核心工件——目标（Goal）、计划（Plan）、观察（Observation）、执行（Execution），正好对应"我要做什么 / 我打算怎么做 / 我看到了什么 / 我做了什么"四个不同语义层级的产物。当 Planner 把候选工具写入 Plan、Explorer 把证据写入 Observation、Executor 把动作写入 Execution，每个 Agent 的提示词中只包含其所需的最小上下文，极大降低 token 开销并消除跨角色串扰。

**编排模式的选择**是另一个关键设计点。MOps 没有采用静态的工作流（Workflow），也没有采用纯粹的对话式 Group Chat，而是选择了"**基于协调器的动态状态机编排**"。状态机由 $\texttt{INIT}\to\texttt{ROUTE}\to\texttt{PLAN}\to\texttt{EXPLORE}\to\texttt{EXECUTE}\to(\texttt{PAUSE}\to\texttt{RESUME})\to\texttt{VERIFY}\to\texttt{SUMMARIZE}\to\texttt{END}$ 九个阶段构成。Coordinator 每轮评估 MASState 决定阶段转移：首轮无计划时转入 PLAN；存在待确认动作且用户刚授权时进入 RESUME 收束；写参就绪且证据充分时进入 EXECUTE。状态机有两类转移边：一类是由当前工件是否完整决定的"正向推进"实线箭头，另一类是由 Verifier 裁决驱动的"回退分派"虚线箭头。

回退分派是 MOps 区别于一般 Plan-Execute 编排的关键能力，对应四种入口：当 Verifier 判定 $\texttt{missing\_evidence}$ 且存在未决问题时，回退到 EXPLORE 补充证据；当判定 $\texttt{risk}$ 时，清空 Plan 与 actions 触发 replan；当判定 $\texttt{retry}$ 且错误可恢复时，把错误动作复位为 pending 触发重试；当同一角色连续两轮无进展时，切换模型或降级到 Plan-Execute 模式触发 reassign。线性 Plan-Execute 编排只能"做完一遍交差"，而 MOps 允许"做了一遍发现不对再做一遍"，这是它在复杂诊断任务上能够保持高准确率的关键。第 5 章消融实验也将印证：取消 Verifier 会让简单任务略快，但让复杂诊断任务的根因命中率显著下降。

为防止长链 Agent 系统的两大隐患——死循环与停滞，Coordinator 同时部署三道守护：硬上界 $\mathrm{lead\_max\_rounds}$ 防止死循环；连续无进展计数 $\mathrm{no\_progress\_count}$ 防止"重复绕圈但每轮内容一致"的状态停滞；工具签名集合 $\mathrm{attempted\_tool\_signatures}$ 防止 LLM 对同一工具用同一参数反复重试。三道守护合在一起，给整个状态机加了一个总时间上界，使得线上稳态运行成为可能。

## 3.4 记忆与上下文管理

### 3.4.1 多源异构数据与上下文建模

MOps 需要整合的数据来源至少包括六类：**用户画像**（身份、角色、权限、历史偏好）、**会话上下文**（多轮对话历史与交互状态）、**页面上下文**（用户当前所看的对象与页面路径）、**任务状态**（进行中任务的中间状态与依赖）、**工具返回**（来自平台 API 的实时切片）、**平台元数据**（作业、资源、告警的静态信息）。这六类数据原始形态千差万别——有的是 JWT 字段，有的是 PostgreSQL 行，有的是 Prometheus 时序，有的是 Kubernetes 事件。要让多个 LLM 角色协同工作，必须把它们统一到一个语义一致的结构化对象上。

MOps 采用一个名为 MASState 的状态对象承担这一统一建模职责。它把六类异构数据归并为五组字段：四阶段工件（Goal/Plan/Observation/Execution）、运行控制（轮次、无进展计数、停止原因、工具签名集合）、证据与审计（tool_records、action_history）、确认与恢复（pending_confirmations、resume_context、workflow checkpoint）、使用统计（token 用量、按角色的模型选用）。在工程上 MASState 是一个 dataclass，但在语义上它是"一次多智能体协作的全部状态"。

可以把 Agent 抽象为"有状态的 LLM 结合感知"：LLM 本身是无状态推理器，感知则来自结构化记忆（MASState 投影）与实时观察（工具返回），二者合起来才构成具有持续认知能力的 Agent。这一视角解释了为什么 MOps 并不依赖于 LLM 自身的"长上下文窗口"——它把外部状态显式化、可控化，因而即使在中等上下文容量的模型上也能稳定工作。

为避免把完整 MASState 原封不动交给每个 Agent（既不必要、不安全、也不经济），MOps 为 MASState 引入一个 **StateView** 投影：每个 Agent 在被调用时只看到一个按角色裁剪后的视图。Planner 视图仅含 Goal 与可用能力；Explorer 视图增加 Plan 与已有 Observation；Executor 视图再增加 actions 草稿；Verifier 视图覆盖 Plan、Observation 与 Execution；Coordinator 视图覆盖全部字段，但以轻量摘要形态出现。StateView 同时承担 token 预算分配——按经验法则把约 45% 的预算给 Explorer（证据最丰富）、25% 给 Executor、20% 给 Planner、10% 给 Verifier，遵循"信息进入决策瓶颈处再压缩"的原则。

### 3.4.2 页面、会话与用户上下文

MOps 按生命周期把上下文划分为三层。**用户上下文**最稳定、权威最高，来自后端 JWT 鉴权，包含用户标识、账户标识、角色与可见能力域，作为整轮对话的可信源；任何来自前端字段或 LLM 输出的角色信息均不可覆盖此层。这一约束是 3.5 节零信任过滤的事实前提——不允许 LLM 自称管理员，是防御提示词注入的最低红线。

**页面上下文**是逐请求更新的隐式参数。前端在用户切换页面时把路由、目标对象（作业名 / 节点名 / 存储卷名）、入口动作、时区语言一并上报。最直接的语义是：当用户在"作业 jobX 的详情页"问"为什么失败"时，Agent 无需用户重复说出作业名，因为页面已经告诉它 jobX 就是焦点对象。这种"所见即所问"的交互依赖前后端协同——前端忠实上报，后端忠实注入，Runtime 忠实使用。

**会话上下文**是跨轮持久化的对话历史，由后端 PostgreSQL 表持久化。Runtime 在每次请求时只从后端取最近若干轮历史，并按总 token 预算贪心装载：历史消息按时间逆序加入直至预算耗尽；超长的工具结果在历史中以"头部 + 尾部 + 省略号"截断（典型为头 1200 字符 + 尾 1200 字符，错误结果保留更多文本以利诊断）；非结构化长文交由轻量 LLM 摘要。这一装载逻辑本质上是一个贪心装箱问题，目标是在预算约束下保留信息量最高的轮次。

### 3.4.3 多轮状态与确认恢复机制

部分运维任务必须等待人类干预，无法在单次 HTTP 请求内闭环。MOps 为此引入 **workflow checkpoint** 机制：当 Executor 准备发起高风险工具调用时，框架冻结当前 MASState，将其序列化并持久化到网关层的数据库中，生成唯一的 confirmation_id；Coordinator 通过 SSE 向前端推送 $\texttt{pause}$ 事件，挂起当前处理流；用户在前端确认卡片上做出决策（批准 / 修改 / 拒绝）；用户的决策请求携带 confirmation_id 返回后端，框架从快照唤醒、反序列化恢复 MASState，把决策合并入 actions 列表后从 EXECUTE 阶段继续推进。

这一机制的关键设计在于：**续接是一个全新的 HTTP 请求，而不是某条长连接的"恢复"**。Runtime 在等待期间真正成为无状态服务——它不持有任何与会话有关的内存状态，等同于一个普通请求/响应服务。任何 Runtime 副本都能拾起任何中断的会话，甚至原 Pod 已被回收后由新副本继续推进也完全可行。这是多轮交互稳健（D2）与水平可扩展两个目标能够同时成立的工程实现基础。

## 3.5 跨智算平台适配的工具体系

工具体系是 MOps 跨平台可迁移性的核心。MOps 把每个工具形式化为一个五元组 $T=(\mathrm{name},\mathrm{schema},\mathrm{executor},\mathrm{permission},\mathrm{confirm})$，其中名称、参数 schema、权限元信息与确认元信息均与具体平台无关，仅 $\mathrm{executor}$ 指向一个抽象适配器接口。生产环境使用面向 Crater 后端的适配器，开发环境使用本地适配器，评测环境使用回放快照的 Mock 适配器；迁移到新智算平台时只需要实现新的适配器子类，工具声明、提示词、协作逻辑均无需修改。这种"声明 vs 执行"的分离对应面向对象设计中的接口隔离原则——把 *what* 与 *how* 解耦。

工具按权限语义被分为四大类：**只读查询类**（节点/作业/告警状态、日志读取等），**用户写类**（重启自身作业、修改自身配置等），**管理员写类**（节点下线、配额调整、批量终止等），**巡检专用类**（仅由 Inspection Agent 调用）。框架基于角色与页面上下文形成一份**权限矩阵** $\Pi(r,p)$——给定用户角色 $r$ 与页面上下文 $p$，矩阵给出可见工具集合：

$$
\Pi(r, p) = \{\, t \in \mathcal{T} \mid \mathbb{1}[t.\mathrm{risk} \in \mathcal{L}_r]\cdot \mathbb{1}[\mathrm{admin}(t) \Rightarrow \mathrm{admin}(r)]\cdot \mathbb{1}[t \in \mathcal{W}_p] = 1 \,\}
$$

三个示性函数分别对应**角色风险约束**、**用户权限约束**与**页面白名单约束**。

权限投影由一个名为 SanitizeTools 的过滤算法在每一轮对话的最初被计算一次，其结果作为工具白名单交给 LLM。零信任过滤遵循三条原则：**角色字段仅来自后端 JWT**，忽略 LLM 输出与前端字段，杜绝提示词注入伪造角色；**页面 scope 可收窄不可放大**，管理员在浏览普通用户页面时被自动降级为普通用户视角，避免角色错位导致误触高风险工具；**在 LLM 看见工具描述之前完成裁剪**，从根本上让 LLM"无法请求它根本看不见的工具"，把提示词注入的攻击面釜底抽薪。

## 3.6 安全控制与审计机制

智算平台运维操作的不可逆性——已删除的 checkpoint、已释放的卡时、已驱逐的节点——决定了安全是 MOps 的一等公民。MOps 把安全约束设计为四道串联的闸门，每一道闸门负责一个独立的失败模态。

**闸门一：权限过滤**在 LLM 注入之前完成，对应 3.5 节的 SanitizeTools。其立意是"治未病"——与其在 LLM 输出阶段事后拦截，不如让无权访问的工具根本不进入 LLM 视野。这道闸门同时承担"页面 scope 不可放大"原则，使得即便管理员误操作浏览了普通用户页面，工具集也自动收窄。

**闸门二：操作确认**针对 medium / high 风险工具。系统在执行前向用户推送确认卡片，清晰展示操作类型、目标对象、潜在影响（例如"将停止 jobX 正在进行的 3000 步训练"）与历史相似案例。批量操作支持逐项勾选，避免"全选执行"的认知陷阱。确认机制本质上是 **人机协同（HITL, Human-in-the-Loop）** 的工程化——MOps 不试图替代人类判断，而是把人类判断引入到关键节点。

**闸门三：确认续接**依赖 3.4.3 节的 workflow checkpoint。当用户做出决定后，系统从快照恢复状态、把确认结果合并到 actions 列表、继续执行。续接全过程对 Runtime 是无状态的——任何副本都能拾起任何中断的会话；用户在确认阶段也可手动修正大模型提取错误的参数（例如把删除资源的数目从 10 改回 1），随后系统执行"确认后恢复与续接"安全地将任务推进到完成态。

**闸门四：审计追踪**对所有运维操作（包括被拒绝的尝试）做全链路记录，含发起者、工具名、入参、结果或错误、延时、风险等级、确认 ID 等字段。除事件级日志外，MOps 同时把每轮的 tool_records 与 action_history 整体序列化保留，使得任何一次运维事件都可以"完整回放"——这对事后归因、合规审查、以及第 4 章质量评估模块的真值构造至关重要。

四道闸门联合保障一个朴素的安全不变量：**任意时刻系统在用户视角下能够看到的风险等级，不会高于初始基线**。这一不变量由闸门一与闸门二联合给出实时保障，由闸门四提供事后核查依据。MOps 借鉴 TNR（Transactional No-Regression）思想并将其延伸到智算训练作业的不可逆操作，使得多智能体系统在保留 LLM 灵活性的同时，获得接近传统运维平台的安全等级。

## 3.7 本章小结

本章给出了面向智算平台的多智能体协作框架 MOps 的系统设计。3.1 节通过对资源—作业—用户三个子空间与四类任务的形式化抽象，明确了运维任务建模与设计需求；3.2 节给出四层总体架构与六条相互掣肘的设计目标，确立了"凭据隔离、Runtime 无状态"两条关键约束；3.3 节定义了核心角色与任务型角色，并通过 Coordinator 状态机与四类回退分派构造了具有自纠错能力的多智能体协作机制；3.4 节通过 MASState 与 StateView 把六类异构数据收敛为统一的结构化共享状态，以 token 预算实现按角色裁剪，并通过 workflow checkpoint 支撑多轮续接；3.5 节通过工具声明—执行分离与权限矩阵保障跨平台可迁移与零信任；3.6 节通过四道安全闸门与 Pause-Confirm-Resume 续接机制保障安全不变量。本章的设计将在第 4 章被工程化为运行在 Crater 平台上的具体实现。
