# 第 1 章 绪论

## 1.1 课题背景与意义

### 1.1.1 课题来源

本课题来源于某高校智算平台 Crater 的实际运维实践。Crater 是一个面向科研与教学场景的异构 GPU 集群管理平台，构建于 Kubernetes + Volcano 之上，承载了包括深度学习训练、大模型微调、Jupyter 交互式开发、WebIDE 在线编程与科学计算在内的多类型 AI 作业。平台支持 PyTorch、TensorFlow、DeepSpeed、OpenMPI、KubeRay 等八类作业框架，管理着来自 NVIDIA、华为昇腾、寒武纪等多个厂商的异构 GPU 设备。

随着平台用户规模从最初的数十人增长至数百人，作业日提交量突破千次，平台运维面临严峻挑战。管理员每日需处理大量重复性工单——从 "我的作业为什么失败了" 到 "能不能帮我看看集群还有多少 GPU 可用"，从 "这个错误信息是什么意思" 到 "帮我把这个节点从调度池摘掉"。与此同时，普通用户在遭遇作业失败、资源配置不当、镜像构建错误等问题时缺乏有效的自助排查工具，不得不依赖管理员人工介入。这种"用户—管理员—平台"之间的信息不对称与高运维负载，已成为制约智算平台服务质量与规模化扩展的核心瓶颈。

本课题针对上述问题，提出**MOps（Multi-agent Orchestration for AI Platform Operations，面向智算平台的多智能体运维编排框架）**，以大语言模型（LLM）驱动的多智能体协作方式，同时为普通用户和管理员提供智能化的运维辅助能力。

### 1.1.2 智算平台的兴起与运维演进

智算平台的出现是深度学习与大规模预训练模型蓬勃发展的直接产物。自 2017 年 Transformer 架构提出以来 [1]，大语言模型参数规模以每年近十倍的速度增长：2018 年 BERT-Large 为 3.4 亿参数 [2]，2020 年 GPT-3 达 1750 亿 [3]，2022 年 PaLM 达 5400 亿 [4]，而近两年开源的 DeepSeek-V3 [5]、Llama-3.1-405B [6] 等模型已进入千亿乃至万亿参数区间。Kaplan 等人的缩放定律（Scaling Laws）研究 [7] 进一步证明模型性能与参数量、数据量、计算量之间存在幂律关系，推动了 "更大即更强" 的研究范式。

大模型参数规模的爆炸式增长直接转化为算力需求的激增。GPT-3 的预训练需约 3640 PF-days 算力，相当于 1024 张 V100 GPU 连续训练超一个月；Meta 在训练 Llama-3.1-405B 时使用了 16000 张 H100 GPU 组成的集群 [6]。如此规模的算力消耗已远超单机或小规模集群的承载能力，必须依赖大规模、异构化的分布式智算基础设施。在政策层面，中国"东数西算"工程将全国算力网络纳入国家级战略 [8]，工信部《算力基础设施高质量发展行动计划》进一步明确算力规模目标 [9]。

回顾 IT 运维方法论近二十年的演进，大致经历了五个阶段：传统人工运维（~2010）→ DevOps 与云计算（2010–2015）→ 云原生可观测性（2015–2019）→ AIOps（2016–2022）→ LLM Ops（2022–2024）。自 2024 年起，以 ReAct [10]、Toolformer [11]、Plan-and-Solve [12] 等 LLM 智能体技术为代表，Agentic AIOps 正成为第六阶段的新范式。然而，**现有 Agentic AIOps 工作绝大多数聚焦于微服务、数据库等通用 IT 系统**，针对智算平台与 GPU 集群的专项研究仍几近空白。

> **表 1-1** IT 运维方法论演进阶段

| 阶段 | 时间 | 代表性技术 | 核心特点 |
|---|---|---|---|
| 传统运维 | ~2010 | Shell 脚本, Nagios, Zabbix | 人工响应，被动告警 |
| DevOps | 2010–2015 | IaC, CI/CD, OpenStack | 标准化与自动化 |
| 云原生可观测 | 2015–2019 | K8s, Prometheus, Jaeger, OpenTelemetry | 三大支柱（Metrics/Logs/Traces） |
| AIOps | 2016–2022 | 异常检测, 日志聚类, 根因定位 | ML 方法，任务专用 |
| LLM Ops | 2022–2024 | ChatGPT, 日志摘要, 告警解释 | 单轮 LLM 调用，幻觉问题 |
| Agentic AIOps | 2024–今 | ReAct, Plan-Execute, Multi-Agent | LLM 主动规划与工具调用 |

### 1.1.3 智算平台的运维独特性

智算平台并非传统云平台的简单延伸，在作业、资源与故障三个维度上呈现显著差异。

**作业维度上**，Crater 平台支持八类作业框架（Jupyter、WebIDE、PyTorch、TensorFlow、DeepSpeed、OpenMPI、KubeRay、自定义作业）。按计算目的可分为训练、推理两类；按并行模式可分为单机、分布式；按交互模式可分为批处理、交互式。不同类型作业对调度、存储、网络、可观测的需求截然不同：训练作业对 GPU 集群带宽敏感、运行时长从小时到数周不等；推理服务对尾延迟敏感；交互作业追求快速响应。

**资源维度上**，GPU 市场已不再是 NVIDIA 一家独大——华为昇腾 910/910B、寒武纪思元等国产加速卡相继进入主流智算平台。同一集群内常混布多代次、多型号 GPU，甚至同时存在 GPU 与专用 AI 芯片。除计算资源外，高速互联网络（InfiniBand、RoCE v2）、高性能并行存储（NFS、Lustre、Ceph）以及 GPU 虚拟化能力同样是平台运维必须管理的一等对象。

**故障维度上**，一个典型的分布式训练作业失败可能来源于：GPU 显存溢出（OOM）、NCCL 初始化超时、RDMA 网卡硬件错误、Pod 驱逐（Eviction）、共享存储 I/O 抖动、数据集路径配置错误、CUDA 版本不匹配、用户代码死锁等数十种原因，且常耦合发生。相比微服务场景中以 "调用链—指标—日志" 为核心的 APM 方法论，智算作业的可观测对象涵盖 GPU SM 利用率、HBM 带宽、NCCL 拓扑等大量领域特异信号。

### 1.1.4 研究意义

**学术层面**，本课题首次系统性地将多智能体 LLM 编排框架引入智算平台运维领域，填补了 Agentic AIOps 在 GPU 集群场景的研究空白；提出的任务感知编排、异构 LLM 分层、证据工程等关键技术可为后续研究提供参考。

**工程层面**，本课题基于 Crater 真实运维场景实现了可落地的 MOps 框架原型，支持 79 种运维工具调用、用户与管理员双视角、读写操作分级审批等生产级特性；配套提出了 Crater-Bench 评测基准。

**社会层面**，通过降低智算平台运维门槛，本课题有望让更多高校与科研机构以更低人力成本运营自有智算基础设施，间接支持国产异构算力生态发展。

## 1.2 国内外研究现状

### 1.2.1 智算平台与 GPU 集群管理

GPU 集群管理研究起步于 2018 年前后。Microsoft Research 的 Gandiva [13] 首次提出针对深度学习作业特点的内省式调度。Tiresias [14] 设计了无先验场景的多级反馈队列算法。Pollux [15]（OSDI 2021）基于 goodput 概念实现吞吐—统计效率联合优化。Alibaba 发表于 NSDI 2022 的《MLaaS in the Wild》[16] 公开了 650 万个 GPU 作业的真实 trace，揭示了生产环境中 GPU 集群作业的显著长尾分布、异构资源需求与频繁失败特征。

工业界方面，NVIDIA Base Command [17]、AWS SageMaker HyperPod、华为 ModelArts、阿里云 PAI 灵骏等智算云平台提供了训练生命周期管理，但运维能力主要体现在基础监控与工单系统。开源生态中，Kubeflow [18]、Volcano [19]、Ray [20] 等组件提供了作业编排支持，但缺乏统一的智能运维层。

**已有 GPU 集群研究几乎全部聚焦于"如何调度"，而非"如何运维"。** 这是本课题切入的重要方向。

### 1.2.2 可观测性与 AIOps

AIOps 领域已有较丰富的综述工作。Notaro 等人 [21] 将 AIOps 研究分为故障预防、检测、根因分析与修复四类，指出根因分析与修复仍是难点。Dang 等人 [22] 从工业实践视角总结了大规模 AIOps 部署的七大挑战。

微服务可观测性方面，Prometheus [23]、Jaeger [24]、OpenTelemetry [25] 构建起事实标准。学术研究中，TraceAnomaly [26] 用变分自编码器检测调用链异常，MicroRCA [27] 基于因果图实现微服务根因定位。然而这些工作建立在 "调用链 + HTTP/gRPC 请求" 假设之上，无法直接迁移到以长时 GPU 作业为核心的智算场景。

### 1.2.3 LLM 智能体与多智能体编排

自 Chain-of-Thought [28] 证明了 LLM 的多步推理能力以来，面向工具使用的智能体技术迅速发展。ReAct [10] 提出推理（Reasoning）与行动（Acting）交替进行的范式。Toolformer [11] 研究了 LLM 自学习调用外部 API 的能力。Plan-and-Solve [12] 让 LLM 先生成规划再执行。Reflexion [29] 引入自我反思机制。

在多智能体层面，AutoGen [30] 提供对话式多智能体抽象，MetaGPT [31] 将软件开发团队的角色分工引入 LLM 协作，LangGraph [32] 以有向图描述复杂工作流，CrewAI [33] 强调任务分配与协作。

### 1.2.4 Agentic AIOps 新进展

Microsoft 的 RCACopilot [34] 率先将 LLM 智能体框架用于微服务根因分析。D-Bot [35] 专注于数据库诊断场景。AIOpsLab [36] 和 ITBench [37] 分别提出了 AIOps 智能体评测基准。然而这些工作均面向微服务或数据库场景，针对智算平台的 Agentic AIOps 研究仍属空白。

## 1.3 本文主要工作与贡献

本文的主要贡献包括：

**贡献一：首次系统化将多智能体 LLM 编排应用于智算平台运维**。深入剖析智算运维的特有挑战，提出领域特化的 MOps 框架。

**贡献二：提出五项关键技术创新**：（C1）任务感知的自适应编排——通过 IntentRouter 实现五类任务的三档模式自适应切换；（C2）异构 Agent 栈与 LLM 分层——9 种角色智能体配合主辅模型路由；（C3）上下文工程与证据压缩——分层 Prompt、Token 预算、证据摘取与历史压缩；（C4）权限感知的双路由工具层——79 工具按权限和性能需求智能路由；（C5）验证智能体与审批链——写操作的全流程可信保障。

**贡献三：构建 Crater-Bench 评测基准**。基于 25812 个真实作业构建的标准化评测场景，提供离线可复现的 Mock 后端与丰富的评测指标。

**贡献四：实际部署验证**。MOps 已在 Crater 平台生产环境运行，积累真实使用数据。

## 1.4 论文组织结构

本文共分为六章。第 1 章为绪论，介绍课题背景、研究现状与主要贡献。第 2 章综述相关概念与技术基础。第 3 章系统阐述 MOps 框架的设计思路与关键技术。第 4 章详细介绍框架在 Crater 平台上的工程实现。第 5 章通过 Crater-Bench 评测基准进行系统实验与评估。第 6 章总结全文并展望未来研究方向。
