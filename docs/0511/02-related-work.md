# 第 2 章 相关概念与技术

本章对支撑 MOps 框架的四个核心技术领域进行系统梳理：（1）智算平台与 GPU 集群管理；（2）可观测性与 AIOps；（3）LLM 智能体与工具使用；（4）多智能体编排框架。章末总结现有工作的不足，明确本课题的切入点。

## 2.1 智算平台与 GPU 集群管理

### 2.1.1 GPU 集群调度研究

GPU 集群调度的核心问题是在多用户、多作业竞争 GPU 资源的情境下最大化集群利用率与公平性。Microsoft Research 发表于 OSDI 2018 的 Gandiva [13] 是这一方向的开山之作，其核心观察是深度学习作业具有迭代性与可预测性，调度器可在 mini-batch 边界处进行抢占、迁移与驱逐而不造成显著性能损失。

Tiresias [14]（NSDI 2019）针对用户缺乏训练时间先验估计的实际问题，设计了基于 LAS（Least Attained Service）的多级反馈队列算法。Pollux [15]（OSDI 2021）首次引入 goodput 概念，将吞吐率与统计效率联合优化，支持 batch size 与资源分配的协同自适应调整。Gavel [38] 进一步提供异构 GPU（V100/P100/T4 混合）的统一调度框架。

在大规模生产 trace 分析方面，Alibaba 发表于 NSDI 2022 的《MLaaS in the Wild》[16] 采集了 650 万个真实 GPU 作业记录，揭示的关键规律包括：GPU 资源需求呈显著长尾分布，约 80% 作业仅需 1-2 张 GPU 而 1% 作业需 32 张以上；作业失败率约 20-30%；失败原因中资源不足与用户代码错误各占约一半。

**然而，上述研究几乎全部聚焦于"如何更好地调度"，而非"如何更好地运维"。** 对已调度作业失败后的根因定位、集群健康度审计与用户自助运维等场景缺乏系统性研究。

### 2.1.2 分布式训练框架与编排

大模型训练依赖分布式训练框架提供并行能力。PyTorch Distributed [39] 与 TensorFlow Distribution Strategy 提供基础数据并行；Horovod [40] 引入 ring-AllReduce 算法；DeepSpeed [41] 提出 ZeRO 优化器实现千亿级模型训练；Megatron-LM [42] 引入模型并行、流水线并行与张量并行的混合并行策略；KubeRay [43] 将 Ray [20] 生态带到 Kubernetes。

编排层面，Kubeflow Training Operator [18] 为 PyTorch/TF/MPI 等框架提供 K8s Operator 封装；Volcano [19] 作为 CNCF 孵化项目提供批调度与 gang-scheduling 能力；Koordinator 提供混部与 QoS 保障。这些框架解决了"如何让 AI 作业跑起来"，但对"为什么没跑起来""什么时候能跑起来"等运维性问题支撑不足。

### 2.1.3 商用智算云平台

商业云厂商在智算平台领域投入巨大：NVIDIA Base Command [17] 面向 DGX 客户提供模型开发环境；AWS SageMaker HyperPod 支持大模型训练集群健康检查与自动恢复；华为云 ModelArts [44]、阿里云 PAI 灵骏、百度百舸等国内平台也提供类似能力。这些平台通常包含作业提交控制台、模型仓库、数据集管理、分布式训练模板与基础监控，但**智能化的用户辅助与管理员面自动化运维仍不在主流提供范围内**。

## 2.2 可观测性与 AIOps

### 2.2.1 可观测性技术体系

现代可观测性（Observability）概念由 Sridharan 在《Distributed Systems Observability》中系统提出 [45]，包括指标（Metrics）、日志（Logs）和分布式调用链（Traces）三大支柱。

**指标层**以 Prometheus [23] 为代表，其多维度数据模型与 PromQL 成为事实标准，配套 Grafana 可视化与 AlertManager 告警路由。在 GPU 场景中，NVIDIA DCGM Exporter 可将 GPU SM 利用率、显存占用、温度、功耗等指标暴露为 Prometheus 端点。

**日志层**历经 ELK 栈到 Loki、Fluent Bit 的发展。在智算场景中，日志不仅包括应用日志和系统日志，还包括 NCCL 通信日志、CUDA 运行时日志、容器运行时日志等，其异构性与体量远超传统微服务日志。

**调用链**方面，Jaeger [24]、Zipkin 是主流分布式追踪实现，但在智算平台中，GPU 作业通常不是以 "请求-响应" 模式运行，传统分布式追踪的实用价值有限。

### 2.2.2 AIOps 研究进展

AIOps 概念由 Gartner 于 2016 年首次提出 [46]，倡导以 ML 方法实现异常检测、根因定位与预测性维护。Notaro 等人 [21] 的系统综述将 AIOps 研究分为故障预防、检测、根因分析与修复四类，指出当时主流方法集中在前两类。

代表性工作包括：Lou 等人利用聚类做日志异常检测 [47]；Microsoft DejaVu 做微服务故障定位 [48]；阿里、腾讯等厂商发布各自 AIOps 平台。然而传统 AIOps 模型往往是**任务专用、样本依赖严重**，且难以处理文本语义与复杂推理。在智算场景中，缺乏标注样本和频繁变更的软硬件栈使得传统 AIOps 方法面临泛化挑战。

## 2.3 LLM 智能体与工具使用

### 2.3.1 LLM 推理能力

Chain-of-Thought（CoT）[28] 证明了 LLM 的多步推理能力：通过在 prompt 中加入 "Let's think step by step"，模型能显著提升数学、逻辑等推理任务的表现。Wei 等人 [49] 进一步发现 CoT 在模型参数达到约 100B 时涌现。Tree-of-Thoughts [50] 探索多路径推理与回溯搜索，Graph-of-Thoughts [51] 将推理建模为有向图。

### 2.3.2 工具使用与智能体范式

ReAct [10] 提出将推理与行动交替进行的范式，奠定了当前主流智能体架构的基础：LLM 在每轮产生一个 Thought（推理），然后发出一个 Action（工具调用），观察 Observation（工具结果），循环直至完成。Toolformer [11] 研究了 LLM 自学习调用外部 API 的能力，通过在预训练语料中插入 API 调用标注实现。

OpenAI Function Calling [52] 将工具调用能力作为模型标准接口，允许开发者定义函数签名（名称、描述、参数 schema），LLM 在推理时输出 JSON 格式的工具调用请求。Anthropic 于 2024 年提出的 Model Context Protocol（MCP）[53] 则意图统一工具协议，使不同 LLM 客户端与工具服务器之间可互操作。

在编排层面，Plan-and-Solve [12] 让 LLM 先生成规划再逐步执行，降低了复杂任务中途跑偏的概率。Reflexion [29] 引入自我反思机制，允许智能体在执行失败后回顾轨迹、提炼教训并重试。Self-Refine [54] 通过 LLM 自我反馈与迭代修改提升输出质量。

### 2.3.3 LLM 推理与成本

LLM 的推理成本（按 token 计费）是智能体部署可行性的决定性因素。以通义千问系列为例，Qwen3-235B-Thinking 的输入价格约 4 元/百万 token，输出约 12 元/百万 token；而 Qwen3-Flash 则分别为 0.3 元/百万 token 和 0.6 元/百万 token，成本差距达 15-20 倍。这意味着在智能体系统中，全部使用顶级大模型将导致成本失控，必须引入异构 LLM 路由策略。

## 2.4 多智能体编排框架

### 2.4.1 通用多智能体框架

AutoGen [30]（Microsoft Research）提供了基于对话的多智能体编程抽象，支持 Agent 之间的消息传递、角色扮演与子任务委托。MetaGPT [31] 将软件开发团队的角色分工（产品经理、架构师、工程师、测试）引入 LLM 协作，展示了结构化 SOP 对多智能体协作质量的显著提升。

LangGraph [32]（LangChain）以有向图描述复杂工作流，节点为计算步骤（LLM 调用或工具执行），边为条件转移。LangGraph 支持循环、分支、并行执行与人机交互（Human-in-the-Loop），其状态持久化机制（Checkpointer）使长时间运行的智能体任务具备断点续传能力。

CrewAI [33] 强调角色化任务分配与团队协作。Agno、Dify、Coze 等低代码 Agent 搭建平台也降低了智能体开发门槛。

### 2.4.2 通用框架的局限

上述通用框架在**领域特化、成本控制与可信保障**三方面存在明显不足：

**领域特化不足**：通用框架的 System Prompt 与工具接口未针对特定领域优化，在智算运维等垂直场景中工具选择盲目、推理链冗长。

**成本控制缺失**：多数框架默认所有智能体使用同一 LLM，未提供异构路由、token 预算、证据压缩等成本控制手段。

**可信保障缺失**：通用框架的 Human-in-the-Loop 机制多为简单的 "暂停-确认-继续"，缺乏风险分级、后验证、审计链等运维场景必需的可信机制。

## 2.5 Agentic AIOps 新进展

2024 年以来，LLM 智能体与运维的融合成为研究热点。Microsoft 的 RCACopilot [34] 率先将 ReAct 框架用于 Azure 微服务根因分析，展示了 LLM 智能体在理解告警上下文、查询监控数据、定位根因方面的潜力。

D-Bot [35]（清华大学）专注于数据库诊断，提取了包含 200+ 数据库诊断知识的领域知识库，通过 RAG 注入智能体提示，在 MySQL/PostgreSQL 场景中取得显著优于 DBA 基线的效果。

AIOpsLab [36]（Microsoft/ETH）和 ITBench [37]（ServiceNow）分别提出了结构化的 AIOps 智能体评测基准，覆盖故障注入、指标异常、日志分析等场景。然而，**以上工作均面向微服务或数据库场景，智算平台与 GPU 集群仍缺乏专项研究**。

## 2.6 现有工作的不足与本课题切入点

综合以上综述，现有工作存在以下不足：

1. **智算平台 Agentic AIOps 空白**：GPU 集群调度与分布式训练已有大量研究，但均聚焦于 "调度" 而非 "运维"；Agentic AIOps 已有初步探索，但均面向微服务/数据库场景。
2. **领域知识缺失**：通用框架的 Prompt 与工具体系缺乏 GPU 集群、NCCL 通信、分布式训练等智算领域的专业知识。
3. **成本可行性未验证**：现有 Agentic AIOps 工作多数考虑了效果，但未系统评估异构 LLM 路由的成本节约潜力。
4. **可信机制缺位**：通用多智能体框架缺乏运维场景必需的写操作确认流、风险分级与审计链。

本课题正是针对上述不足，提出面向智算平台的多智能体运维编排框架 MOps。
