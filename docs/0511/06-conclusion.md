# 第 6 章 总结与展望

## 6.1 工作总结

本课题针对智算平台运维的两大群体痛点——用户缺乏自助排查手段、管理员承担高重复性工单负担——设计并实现了面向智算平台的多智能体运维编排框架 **MOps（Multi-agent Orchestration for AI Platform Operations）**，并通过系统实验验证了框架设计的合理性与有效性。

### 6.1.1 主要贡献

**贡献一：首次系统化将多智能体 LLM 编排应用于智算平台运维**。本课题填补了 Agentic AIOps 在 GPU 集群场景的研究空白。相比已有工作多针对微服务或数据库场景，MOps 深入剖析了智算运维的特有挑战——作业异构、资源异构、故障模式复杂——并给出了完整解决方案。

**贡献二：提出并实现了五项关键技术创新**：

- **C1 任务感知的自适应编排**：通过 IntentRouter 实现五类任务的三档模式自适应切换。实验证明在困难诊断场景中 MOPS 相比 React 提升 8-16 分，而在简单场景中三方法持平，验证了自适应编排的设计逻辑。

- **C2 异构 Agent 栈与 LLM 分层**：9 种角色智能体配合主模型（Thinking）与辅助模型（Flash）的异构路由。每次调用的平均 Token 消耗仅为 React 的 42%（1934 vs 4569）。

- **C3 上下文工程与证据压缩**：分层 System Prompt、Token 预算控制、LLM 驱动的证据摘取与历史压缩，有效解决了长轨迹任务的上下文爆炸问题。

- **C4 权限感知的双路由工具层**：79 个工具通过进程内本地执行与 HTTP 远程执行的双路由架构满足高性能与安全隔离需求。角色-工具权限矩阵确保安全边界。

- **C5 验证智能体与审批链**：写操作的风险分级、确认流与后验证机制确保运维操作的可靠性。

**贡献三：构建 Crater-Bench 评测基准**。基于 25812 个真实作业构建的 30 个标准化评测场景，提供离线可复现的 Mock 后端与多维评测指标。实验系统地对比了 MOPS、PS、React 三种方法，并通过 DeepSeek-V4-Pro 扩展验证了框架的跨模型泛化能力。

**贡献四：实际部署**。MOps 已作为 Crater 智算平台的一部分在生产环境运行，服务真实用户。

### 6.1.2 量化结果总结

- 工具选择质量：MOPS 的 Tool-F1 为 0.95，较 React（0.81）提升 17.3%。
- 困难诊断场景：终止类诊断提升 16.07 分（70.36→86.44），OOM 诊断提升 8.30 分（87.98→96.28）。
- 跨模型泛化：DeepSeek-V4-Pro 下 MOPS 取得 93.78 平均分和满分 Tool-F1（1.0）。
- 成本效率：MOPS 每次 LLM 调用的平均 Token 消耗为 1934，仅为 React（4569）的 42%。

## 6.2 工作的局限与未来展望

### 6.2.1 当前局限

**LLM 家族覆盖有限**：本课题实验采用了 Qwen-max 和 DeepSeek-V4-Pro 两个模型家族，未验证在 GPT-4o、Claude、Llama 等更多模型上的表现。

**实验重复性不足**：由于时间和成本限制，当前实验仅对每场景运行 1 个 seed，缺乏多次重复以评估统计显著性。

**Remedy 类场景缺位**：Crater-Bench 首版 30 个场景未覆盖修复（Remedy）类任务，C5（验证智能体）的贡献在实验章中未能充分量化。

**在线验证有限**：实验以离线 snapshot 模式为主，框架在大规模生产负载下的表现尚需进一步观察。

### 6.2.2 未来工作

1. **Crater-Bench v2**：扩展评测场景至 60+，特别覆盖 Remedy 场景（节点驱逐、作业重启、配额调整）、多轮对话场景与紧急响应场景。
2. **跨家族 LLM 验证**：在 Qwen3、Claude、GPT-4o、Llama 等至少 4 个 LLM 家族上运行主实验。
3. **领域知识自动内化**：将历史工单中的管理员故障处理 SOP 构建为结构化知识库，通过 RAG 动态注入智能体推理。
4. **在线学习与反馈闭环**：构建从用户反馈到智能体改进的闭环，利用已有的 `agent_feedbacks` 和 `agent_quality_evals` 表实现 LLM-as-Judge 评估到自动调优。
5. **多模态拓展**：支持 GPU 拓扑图、资源使用曲线等多模态输入，使智能体能直接阅读和生成 Grafana 图表。
6. **主动预测服务**：从响应式运维走向预测式运维——在故障发生前主动预警，在资源紧张前主动建议。

## 6.3 结语

智算平台的普及是 AI 产业发展的基础设施层面必经之路，但高昂的使用门槛与运维负担正在制约其规模化发展。LLM 智能体技术为此带来了新的可能，但通用框架不足以应对智算场景的独特挑战。MOps 通过任务感知编排、异构 LLM 分层、证据工程、权限感知工具层与验证链，展示了面向智算平台的领域特化多智能体框架的可行性与价值。

"MOps" 之名蕴含两层含义：既是 Multi-agent Orchestration for AI Platform Operations 的首字母缩写，也如英文 "mop"（拖把）一般，意在扫除智算平台用户与管理员运维道路上的障碍。希望本课题的思想与实践能为后续智算平台运维研究提供一个可供参考的基线，也能真正为 Crater 等真实平台的用户和管理员带来便利。
