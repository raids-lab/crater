# 第 3 章 MOps 多智能体运维编排框架的设计

本章是论文的核心技术章节，系统阐述面向智算平台的多智能体运维编排框架 MOps 的设计理念与关键技术。3.1 节给出总体架构，3.2 至 3.6 节分别展开五项关键创新设计。

## 3.1 总体架构

### 3.1.1 设计原则

MOps 框架的设计遵循以下五项原则：

**原则 1：领域特化优先（Domain-Specialized First）**。与 AutoGen、MetaGPT 等通用多智能体框架不同，MOps 从设计之初就针对智算平台运维场景做深度定制：工具层覆盖 GPU 集群专用诊断能力，System Prompt 融入分布式训练领域知识，任务分类对齐平台实际用户行为模式。

**原则 2：异构分层（Heterogeneous Layering）**。拒绝"一个大模型打天下"的假设。在 Agent 层面按能力需求混用思考型大模型与轻量快速模型；在编排层面按任务复杂度切换不同编排模式；在工具层面根据权限和性能需求采用进程内调用或远程 HTTP 调用。

**原则 3：成本感知（Cost-Aware）**。承认 LLM 调用的 token 成本是部署可行性的决定性因素。通过异构 LLM 路由、证据压缩、上下文管理与实时 token 计费等手段控制成本，使单次智能体任务的平均成本控制在可接受区间。

**原则 4：可信优先（Trust-First）**。运维场景下，一次错误的节点驱逐或作业删除可能造成严重损失。所有写操作必须经过显式确认，必须有可追溯的完整审计日志，必须经过后验证。这是 MOps 与通用多智能体框架最重要的差异。

**原则 5：平台解耦（Platform-Agnostic）**。尽管 MOps 最初基于 Crater 平台实现，但其架构保证与具体平台解耦。工具定义与执行分离、Prompt 与运行时分离、Agent 角色与实现分离，确保框架能够移植到其他智算平台。

### 3.1.2 四层架构

MOps 框架采用四层架构，如图 3-1 所示。

> **图 3-1** MOps 框架总体架构图
> 
> 四层架构说明：
> - **用户交互层**：Web Chat（普通用户）、Admin Console（管理员）、外部 API
> - **编排层**：IntentRouter → Coordinator → ReAct / Plan-Execute / Plan-Execute-Verify
> - **Agent 层**：9 种角色智能体（Coordinator/Planner/Explorer/Executor/Verifier/Approval/GeneralPurpose/Guide/MainAgent）
> - **工具层**：79 个工具，8 个类别，进程内本地执行 + HTTP 远程执行双路由

```
┌─────────────────────────────────────────────────────┐
│                   用户交互层                          │
│   Web Chat (SSE 流式)  │  Admin Console  │  API      │
├─────────────────────────────────────────────────────┤
│                   编排层 (Orchestration)              │
│  ┌──────────┐   ┌──────────┐   ┌────────────────┐  │
│  │IntentRouter│→ │Coordinator│→  │ReAct / Plan-   │  │
│  │ 任务分类   │   │ 调度协调   │   │Execute / PV    │  │
│  └──────────┘   └──────────┘   └────────────────┘  │
├─────────────────────────────────────────────────────┤
│                   Agent 层 (Agent Layer)             │
│  Planner │ Explorer │ Executor │ Verifier │ ...     │
│  (Flash) │ (Flash)  │ (Think)  │ (Think)  │         │
├─────────────────────────────────────────────────────┤
│                   工具层 (Tool Layer)                 │
│  作业诊断 │ 指标队列 │ 集群管理 │ 存储数据 │ ...     │
│  本地执行 (高性能)  ←→  HTTP 远程 (安全隔离)        │
└─────────────────────────────────────────────────────┘
```

**上层：用户交互层**。面向普通用户的 Web 聊天窗口，面向管理员的管理控制台，以及对接外部审批流程的内部 API。通过 SSE（Server-Sent Events）以流式方式向前端推送智能体的推理过程（Thought）、工具调用事件（Tool Call/Result）与最终结果。

**中上层：编排层（Orchestration Layer）**。核心包含意图路由器（IntentRouter）、协调器（Coordinator）和三档编排模式。IntentRouter 负责将用户请求分类为查询、诊断、运维审计、工单提交、修复五大类。Coordinator 根据分类结果选择编排模式并调度 Agent 层执行。运行时基于 LangGraph 实现，所有节点过渡可持久化以支持重连与审批等待。

**中下层：Agent 层（Agent Layer）**。由九种角色智能体组成：协调器（Coordinator）、规划器（Planner）、探索器（Explorer）、执行器（Executor）、验证器（Verifier）、审批器（Approval）、通用智能体（GeneralPurpose）、向导智能体（Guide）和主智能体（MainAgent）。每个角色有三个核心配置要素：System Prompt、LLM 模型配置、可用工具范围。所有角色由 `BaseRoleAgent` 类统一抽象。

**底层：工具层（Tool Layer）**。覆盖 79 种工具，分为作业诊断、指标与队列、集群管理、存储与数据、GPU 与网络、作业操作、审计与审批、K8s 原生八大类别。通过 Composite 模式实现进程内本地执行（高性能）与 HTTP 远程执行（安全隔离）的智能路由。

### 3.1.3 典型请求处理流程

一个典型用户请求的处理流程如下：

```
[用户输入] "我的作业 job-xyz 为什么失败了？"
      ↓
[意图路由] IntentRouter 识别为诊断类 (confidence: 0.95)
      ↓
[模式选择] Coordinator 选择 Plan-Execute 模式
      ↓
[规划阶段] Planner (Flash 模型, 低成本)
   Step 1: 查作业详情 (get_job_detail)
   Step 2: 查作业事件 (get_job_events)
   Step 3: 查作业日志 (get_job_logs)
   Step 4: 综合分析 (diagnose_job)
      ↓
[探索阶段] Explorer (Flash 模型) 依次执行 Steps 1-3
      ↓
[执行阶段] Executor (Thinking 模型) 基于收集的证据进行诊断推理
      ↓
[最终输出] 返回给用户格式化的诊断报告
```

## 3.2 创新点 C1：任务感知的自适应编排

### 3.2.1 任务五分类体系

MOps 将智算平台运维任务分为五大类：

| 类别 | 典型请求 | 关键特征 | 推荐模式 |
|---|---|---|---|
| **查询 (Query)** | "集群还有几张 A100？" | 单步工具调用，无推理需求 | ReAct（最小开销） |
| **诊断 (Diagnosis)** | "为什么我的作业 OOM 了？" | 多步工具调用与推理 | Plan-Execute |
| **运维审计 (Ops Audit)** | "检查集群所有节点健康状态" | 大量数据收集与汇总 | Plan-Execute |
| **工单提交 (Submission)** | "帮我提交一个 4 卡训练作业" | 需要确认流与配额检查 | Plan-Execute-Verify |
| **修复 (Remedy)** | "把 gpu-node-09 从调度池摘除" | 写操作，需审批与后验证 | Plan-Execute-Verify |

### 3.2.2 IntentRouter 设计

IntentRouter 采用关键词匹配 + LLM 分类的两阶段策略。第一阶段通过关键词快速匹配（如 "失败""OOM""报错" → 诊断类），置信度低于阈值时进入第二阶段——由轻量 LLM 进行语义分类。

分类结果不仅决定编排模式，还影响后续 Agent 的 System Prompt 注入：诊断类任务注入故障排查指南，提交类任务注入模板匹配与配额检查逻辑。

### 3.2.3 三档编排模式

**模式 1：ReAct（推理-行动循环）**。单智能体交替执行推理与工具调用。适用于查询类简单任务。优势是最小 LLM 调用开销，缺点是无法进行复杂的多步规划。

**模式 2：Plan-Execute（规划-执行）**。先由 Planner（Flash 模型）生成结构化调查计划，再由 Explorer 执行只读工具收集数据，最后由 Executor（Thinking 模型）基于完整证据进行深度推理。适用于诊断与审计类任务。

**模式 3：Plan-Execute-Verify（规划-执行-验证）**。在 Plan-Execute 基础上增加 Verifier 对写操作结果进行后验证。适用于工单提交与修复类高风险任务。Verifier 通过对比写操作前后的平台快照（如节点调度状态、作业运行状态），确认操作结果符合预期。

## 3.3 创新点 C2：异构 Agent 栈与 LLM 分层

### 3.3.1 九种角色智能体

MOps 定义了九种角色智能体，各司其职：

| 角色 | 职责 | LLM 选择 | 工具权限 |
|---|---|---|---|
| Coordinator | 任务分解与 Agent 调度 | Flash | 无工具 |
| Planner | 结构化调查计划生成 | Flash | 无工具 |
| Explorer | 只读数据收集与证据整理 | Flash | 只读工具 |
| Executor | 深度推理与最终输出 | Thinking | 只读工具 |
| Verifier | 操作后验证与合规检查 | Thinking | 读 + 审计工具 |
| Approval | 审批流程管理 | Flash | 审批工具 |
| GeneralPurpose | 通用问答与兜底 | Flash | 全部只读 |
| Guide | 向导式交互引导 | Flash | 只读 |
| MainAgent | 单 Agent ReAct 模式主体 | Flash | 按角色 |

### 3.3.2 异构 LLM 路由策略

MOps 采用"主模型 + 辅助模型"的异构路由策略：

- **主模型（Thinking）**：Qwen3-235B-Thinking，用于 Executor、Verifier 等需要深度推理的角色。价格约输入 4 元/百万 token，输出 12 元/百万 token。
- **辅助模型（Flash）**：Qwen3-Flash，用于 Planner、Explorer、Coordinator 等执行结构化任务的中间步骤。价格约输入 0.3 元/百万 token，输出 0.6 元/百万 token。

路由决策依据任务阶段而非用户可见的最终结果质量。规划（Plan）、探索（Explore）、意图路由（IntentRoute）等工作由 Flash 完成，只有最终的诊断推理、验证判断由 Thinking 完成。这种分层使 Thinking 模型聚焦于其真正擅长的深度推理，而将"体力活"卸载给低成本模型。

## 3.4 创新点 C3：上下文工程与证据压缩

### 3.4.1 分层 System Prompt

MOps 的 System Prompt 分为三层：

- **L1 全局层**：角色定义、行为准则、安全规范（约 500 token）
- **L2 领域层**：智算平台运维知识、故障模式库、诊断 SOP（约 1500 token）
- **L3 任务层**：当前任务上下文、用户信息、可用工具清单（动态注入）

分层设计使得不同角色可以共享 L1/L2 层，仅切换 L3 层。诊断类任务额外注入故障排查指南，提交类任务额外注入模板匹配逻辑。

### 3.4.2 Token 预算与证据摘取

长轨迹任务的最大挑战是上下文爆炸：当 Explorer 调用 5-8 个工具后，累计的工具返回结果可能达到 10000-20000 token，远超 LLM 上下文窗口的有效使用区间。

MOps 的解决方案是**证据摘取（Evidence Extraction）**：Explorer 在收集完数据后，调用一次额外的 Flash LLM 调用，将原始工具输出压缩为结构化证据摘要（仅保留关键数据字段与分析线索），再传递给 Executor。这使 Executor 接收的上下文从原始 10000+ token 压缩至 2000-3000 token。

此外，MOps 维护一个全局 token 计数器，当累计 token 接近预算上限时自动触发会话历史压缩——保留最近 N 轮完整交互，将更早的轮次替换为 LLM 生成的摘要。

## 3.5 创新点 C4：权限感知的双路由工具层

### 3.5.1 工具分类与权限矩阵

MOps 的 79 个工具按功能分为八大类，每类分配不同的权限策略：

| 类别 | 数量 | 权限策略 | 路由方式 |
|---|---|---|---|
| 作业诊断 | 8 | 只读，用户可调用 | 本地 |
| 指标与队列 | 12 | 只读，用户可调用 | 本地 |
| 集群管理 | 15 | 只读/读写，管理员可调用 | HTTP 远程 |
| 存储与数据 | 6 | 只读，用户可调用 | HTTP 远程 |
| GPU 与网络 | 10 | 只读，用户可调用 | 本地 |
| 作业操作 | 9 | 读写，需确认 | HTTP 远程 |
| 审计与审批 | 7 | 读写，需审批 | HTTP 远程 |
| K8s 原生 | 12 | 只读/读写，管理员 | HTTP 远程 |

每个角色智能体在实例化时被赋予一个工具白名单：Explorer 只能访问只读诊断工具，Executor 只能访问只读工具但范围更广，Approval 可以访问审批工具。

### 3.5.2 双路由机制

**进程内本地执行**：对于高频调用且无安全风险的只读工具（如 `get_job_detail`、`analyze_queue_status`），直接在 Agent 进程内通过 Mock 或本地函数调用执行。响应时间为毫秒级，零网络开销。

**HTTP 远程执行**：对于涉及真实平台操作的读写工具（如 `cordon_node`、`delete_job`）和需要后端数据查询的工具，通过 HTTP 调用 Crater 后端 API。请求携带 Agent 的身份认证信息（API Key），后端进行二次权限校验。

路由决策由工具注册时的元数据（`execution_mode: local | remote`）决定，Agent 代码无需感知执行位置。

## 3.6 创新点 C5：验证智能体与审批链

### 3.6.1 风险分级

MOps 将所有写操作按风险分为三级：

- **低风险**：影响单个资源的查询性写操作（如提交新作业）。需要用户显式确认。
- **中风险**：影响已有作业的操作（如停止作业、修改配额）。需要用户确认 + Verifier 后验证。
- **高风险**：影响集群节点的操作（如节点驱逐、调度池变更）。需要管理员审批 + 用户确认 + Verifier 后验证。

### 3.6.2 确认流与后验证

对于写操作，MOps 在 Executor 生成操作计划后暂停，向前端推送确认请求。确认请求包含：操作描述、影响范围、风险评估、Pre-check 结果（如配额检查、节点状态）。用户确认后，Executor 执行操作，Verifier 立即进行后验证——通过对比操作前后的平台状态快照，确认结果符合预期。

### 3.6.3 审计链

所有操作（包括读操作）均记录完整的调用链日志，包括：时间戳、操作用户、Agent 角色、LLM 模型、工具调用参数与返回值、确认记录、验证结果。审计日志存储在 PostgreSQL 中，支持按用户、时间、操作类型检索，满足事后追溯与合规审查需求。
